# 運用手順

本書は、`11_インフラストラクチャ設計.md`が定義する起動・ヘルスチェック・Graceful
Shutdownの設計を、実装（`src/bootstrap/index.ts`・`src/presentation/http/health-routes.ts`・
`src/bootstrap/shutdown.ts`）がどう満たすかをまとめる。詳細な設計判断は
`11_インフラストラクチャ設計.md`「ヘルスチェック」「Graceful Shutdown」「ログ設計」を参照する。

## 起動

1. `.env.example`を参考に環境変数を設定する（`PORT`・`CATALOG_PATH`・`LOG_LEVEL`・
   `SHUTDOWN_GRACE_MS`など）。
2. `mise run build`でTypeScriptをコンパイルする。
3. `node dist/main.js`で起動する（`mise run dev`は開発時の`tsx watch`版）。
4. `src/bootstrap/index.ts`の`bootstrap()`が次を順に行う。
   - Catalog manifestから期待する`catalogRevision`を読む。
   - `SimulationWorkerPool.create()`でWorkerをwarm-upし、Catalogの読み込み・
     検証・リビジョン一致を確認する（失敗時はここで例外を投げ、HTTPポートを
     一切公開しない）。
   - `buildServer()`でFastifyインスタンスを構築し、`/health/live`・
     `/health/ready`・`POST /api/v1/battle-simulations`を登録する。
   - SIGTERM/SIGINTハンドラーを登録する（`installShutdownSignalHandlers`）。
   - `listen()`してから起動ログ（`catalogRevision`・`workerMaxQueue`・
     `simulationTimeoutMs`）を1行出す。
5. Worker初期化に失敗した場合、`bootstrap()`は例外をrejectし、`main.ts`が
   `process.exit(1)`する。ポートは開かれない。

## ヘルスチェック

| エンドポイント      | 成功条件                                                                                 | 用途                                     |
| ------------------- | ---------------------------------------------------------------------------------------- | ---------------------------------------- |
| `GET /health/live`  | プロセスがHTTP応答可能であること。Catalog障害やPool飽和では失敗しない。                  | プロセスの生存確認（再起動要否の判定）。 |
| `GET /health/ready` | シャットダウン未開始、かつWorker Poolが健全（稼働中のCatalogリビジョン不一致が未発生）。 | ロードバランサーへのトラフィック可否。   |

`/health/ready`は`503`を返す際も`{"status":"not_ready"}`のような最小bodyのみを返す
——Catalogの中身・環境変数・エラーのスタックは含めない。

起動シーケンス上、`listen()`に到達する時点で必要ワーカー数のwarm-upと
Catalog検証は既に完了しているため、「準備未完了」を外部から観測できるのは
実質的に次の2状態だけになる。

- Graceful Shutdown開始後（`ShutdownState.markShuttingDown()`）。
- 稼働中にWorkerがCatalogリビジョン不一致を報告し、Pool全体が致命的状態に
  なった後（`SimulationWorkerPool.isHealthy`が`false`）。

## Graceful Shutdown

SIGTERMまたはSIGINTを受信すると、`runGracefulShutdown`が次を行う。

1. `ShutdownState.markShuttingDown()`——`/health/ready`が即座に失敗を返すようになる。
2. 以後の`POST /api/v1/battle-simulations`はUseCaseへ到達せず、
   `503 CAPACITY_EXCEEDED`（`Retry-After`付き）を返す。
3. `app.close()`（HTTP keep-alive接続のdrain）と`pool.shutdown()`
   （Worker Poolの停止）を同時に開始する。
   - `pool.shutdown()`は未開始タスクを即座にキャンセルし、実行中タスクを
     `SHUTDOWN_GRACE_MS`まで待ってから、残っていれば強制終了する
     （どちらの経路でも`EXECUTION_CANCELLED`として応答する）。
   - `app.close()`は、対応するHTTPハンドラーが完了・rejectし次第、drainを終える。
4. 両方の完了を待って`process.exit(0)`する。シャットダウン処理自体が失敗した
   場合はエラーをログへ記録し`process.exit(1)`する。

ロードバランサーのdrain猶予は`SHUTDOWN_GRACE_MS`より長く設定する。戦闘結果を
永続化しないため、シャットダウン中に中断した結果は復旧しない。

## 構造化ログ

`LOG_LEVEL`で有効化されるFastify内蔵のpinoロガーがJSON形式で出力する。

- 全リクエストで`requestId`フィールドが自動で付く（`requestIdLogLabel`設定と、
  `X-Request-Id`ヘッダーまたは生成UUIDを解決する`genReqId`による）。
- 戦闘完了時に`catalogRevision`・`battleId`・`outcome`・`completionReason`・
  `completedTurn`・`eventCount`・`stateTransitionCount`を1行で記録する。
- 500番台のエラーは`diagnosticId`をレスポンスとログの両方へ記録し、事後に
  サーバーログと照合できるようにする。

生のリクエスト・レスポンス本文やCatalog全体は記録しない
（`11_インフラストラクチャ設計.md`「ログ設計」参照）。
