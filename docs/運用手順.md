# 運用手順

本書は、`11_インフラストラクチャ設計.md`が定義する起動・ヘルスチェック・Graceful
Shutdownの設計を、実装（`apps/api/src/bootstrap/index.ts`・
`apps/api/src/presentation/http/health-routes.ts`・
`apps/api/src/bootstrap/shutdown.ts`）がどう満たすかをまとめる。詳細な設計判断は
`11_インフラストラクチャ設計.md`「ヘルスチェック」「Graceful Shutdown」「ログ設計」を参照する。

## M4.5配備構成

M4.5ではAPIをGoogle Cloud Run serviceへ配備する。初期構成は次とする。

| 項目                  | 値                              |
| --------------------- | ------------------------------- |
| Service               | `muvluvgg-battle-simulator-api` |
| Region                | `asia-northeast1`               |
| Container registry    | Google Artifact Registry        |
| CPU／Memory           | 1 vCPU／1 GiB                   |
| Minimum／Maximum      | 0／1 instance                   |
| Container concurrency | 2                               |
| Request timeout       | 40秒                            |
| Invocation            | public。Cloud Run IAM認証なし。 |
| Browser origin        | `https://komei0727.github.io`   |

GitHub Pagesからの初期アクセスではscale-to-zero後のcold startが発生し得る。Catalog読込とWorker warm-upが完了してからlistenする既存起動順を維持し、不完全なreadinessでrequestを受け付けない。

## 起動

1. `.env.example`を参考に環境変数を設定する（`PORT`・`CATALOG_PATH`・`LOG_LEVEL`・
   `SHUTDOWN_GRACE_MS`など）。
2. `mise run build`でTypeScriptをコンパイルする。
3. `node apps/api/dist/main.js`で起動する（`mise run dev`は開発時の`tsx watch`版）。
4. `apps/api/src/bootstrap/index.ts`の`bootstrap()`が次を順に行う。
   - Catalog manifestから期待する`catalogRevision`を読む。
   - `SimulationWorkerPool.create()`でWorkerをwarm-upし、Catalogの読み込み・
     検証・リビジョン一致を確認する（失敗時はここで例外を投げ、HTTPポートを
     一切公開しない）。
   - `buildServer()`でFastifyインスタンスを構築し、`/health/live`・
     `/health/ready`・`GET /api/v1/battle-simulation-catalog`・`POST /api/v1/battle-simulations`を登録する。
   - SIGTERM/SIGINTハンドラーを登録する（`installShutdownSignalHandlers`）。
   - `listen()`してから起動ログ（`catalogRevision`・`workerMaxQueue`・
     `simulationTimeoutMs`）を1行出す。
5. Worker初期化に失敗した場合、`bootstrap()`は例外をrejectし、`main.ts`が
   `process.exit(1)`する。ポートは開かれない。

## OpenAPI文書とSwagger UI

`11_インフラストラクチャ設計.md`「OpenAPI」: `GET /openapi.json`は常に生成済みの
OpenAPI 3.0.3文書を返す。Swagger UI（`GET /docs`）は`NODE_ENV`が`production`
以外の場合だけ`bootstrap()`が有効化する——productionでは既定で公開しない
（`apps/api/src/bootstrap/index.ts`の`docsEnabled`、
`apps/api/src/presentation/http/build-server.ts`の`BuildServerOptions.docsEnabled`）。

1. `mise run dev`で起動する（`.env.example`の既定`NODE_ENV=development`のまま）。
2. ブラウザで `http://localhost:3000/docs` を開くとSwagger UIが表示される。
   `POST /api/v1/battle-simulations`の仕様を確認し、UIから最小戦闘リクエストを
   実行できる。
3. `http://localhost:3000/openapi.json` はSwagger UIの有効・無効に関わらず、
   同じOpenAPI文書を返す。
4. `NODE_ENV=production`で起動した場合、`GET /docs`は`404`を返す。

## ヘルスチェック

| エンドポイント      | 成功条件                                                                                 | 用途                                     |
| ------------------- | ---------------------------------------------------------------------------------------- | ---------------------------------------- |
| `GET /health/live`  | プロセスがHTTP応答可能であること。Catalog障害やPool飽和では失敗しない。                  | プロセスの生存確認（再起動要否の判定）。 |
| `GET /health/ready` | シャットダウン未開始、かつWorker Poolが健全（稼働中のCatalogリビジョン不一致が未発生）。 | ロードバランサーへのトラフィック可否。   |

`/health/ready`は`503`を返す際も`{"status":"not_ready"}`のような最小bodyのみを返す
——Catalogの中身・環境変数・エラーのスタックは含めない。

起動シーケンス上、`listen()`に到達する時点で必要ワーカー数のwarm-upと
Catalog検証は既に完了しているため、「準備未完了」を外部から観測できるのは
実質的に次の2状態だけになる。

- Graceful Shutdown開始後（`ShutdownState.markShuttingDown()`）。
- 稼働中にWorkerがCatalogリビジョン不一致を報告し、Pool全体が致命的状態に
  なった後（`SimulationWorkerPool.isHealthy`が`false`）。

## Graceful Shutdown

SIGTERMまたはSIGINTを受信すると、`runGracefulShutdown`が次を行う。

1. `ShutdownState.markShuttingDown()`——`/health/ready`が即座に失敗を返すようになる。
2. 以後の`POST /api/v1/battle-simulations`はUseCaseへ到達せず、
   `503 CAPACITY_EXCEEDED`（`Retry-After`付き）を返す。
3. `app.close()`（HTTP keep-alive接続のdrain）と`pool.shutdown()`
   （Worker Poolの停止）を同時に開始する。
   - `pool.shutdown()`は未開始タスクを即座にキャンセルし、実行中タスクを
     `SHUTDOWN_GRACE_MS`まで待ってから、残っていれば強制終了する
     （どちらの経路でも`EXECUTION_CANCELLED`として応答する）。
   - `app.close()`は、対応するHTTPハンドラーが完了・rejectし次第、drainを終える。
4. 両方の完了を待って`process.exit(0)`する。シャットダウン処理自体が失敗した
   場合はエラーをログへ記録し`process.exit(1)`する。

ロードバランサーのdrain猶予は`SHUTDOWN_GRACE_MS`より長く設定する。戦闘結果を
永続化しないため、シャットダウン中に中断した結果は復旧しない。

Cloud RunはSIGTERM後10秒でcontainerを強制終了し得るため、productionの `SHUTDOWN_GRACE_MS`は8000とする。8秒以内に完了しない実行中simulationはキャンセルし、不完全な戦闘結果を成功として返さない。

## Production ContainerとCloud Run service基盤

`#105`（`M45-INFRA-001`）が次を用意する。CI/CDへの配線（Artifact Registry push、
Cloud Run deploy、post-deploy smoke test、rollback）は`#106`（`M45-INFRA-002`）
のスコープ。

- [`Dockerfile`](../Dockerfile)：multi-stage build。production依存
  （`pnpm install --frozen-lockfile --prod`）、`tsc`でコンパイルした
  `apps/api/dist/`、`catalog/`（`CATALOG_PATH`の既定値に合わせ`/app/catalog`へ
  flattenして含める）だけをruntime imageへ含める。`node@sha256:...`
  （`node:24.18.0-bookworm-slim`、mise.tomlのNode.jsバージョンと一致）を
  `--platform=linux/amd64`で固定し、non-root user（`nodeapp`、uid/gid 1001）
  で`CMD ["node", "--enable-source-maps", "apps/api/dist/main.js"]`を実行する。
- [`.dockerignore`](../.dockerignore)：`node_modules`・`dist`・`docs`・`raw`・
  `catalog-src`など、runtime imageに不要なファイルをbuild contextから除外する。
- [`deploy/cloud-run/service.json`](../deploy/cloud-run/service.json)：
  「M4.5配備構成」の初期値（CPU/Memory、min/max instance、concurrency、
  timeout、`WORKER_MAX_QUEUE`、`SHUTDOWN_GRACE_MS`、CORS origin）を宣言する
  Knative Service manifest（`gcloud run services replace`が読む形式）。
  値は`11_インフラストラクチャ設計.md`「M4.5 Cloud Run初期設定」の理由付き表を
  正本とし、[`apps/api/src/__tests__/cloud-run-service-config.test.ts`](../apps/api/src/__tests__/cloud-run-service-config.test.ts)
  が自動で一致を検証する。適用コマンドと`region`/`project`の渡し方は
  [`deploy/cloud-run/README.md`](../deploy/cloud-run/README.md)を参照。
- `mise run test:container`（[`scripts/container-smoke-test.sh`](../scripts/container-smoke-test.sh)、
  Docker必須）：image build、non-root起動、`PORT`上書き・`0.0.0.0` bind、
  baked-inのCatalog・Worker・OpenAPI解決、`/docs`が本番で404、SIGTERM後の
  graceful shutdown（`SHUTDOWN_GRACE_MS=8000`以内・exit code 0）、
  bind-mountしたfixture Catalogでの最小simulation成功をローカルで検証する
  （`mise run check`には含めない——`test:integration`/`test:e2e`/`test:load`
  と同様、Dockerを要求するため個別実行する）。

Artifact Registry repositoryは事前に一度だけ作成する。

```bash
gcloud artifacts repositories create muvluvgg-battle-simulator \
  --repository-format=docker \
  --location=asia-northeast1 \
  --project="$PROJECT_ID" \
  --description="muvluvgg-battle-simulator production container images"
```

## Cloud Run deploy

1. PRでformat、typecheck、lint、test、production build、container build、local container smoke testを成功させる。
2. main workflowがGitHub OIDC tokenを使い、Workload Identity FederationでGoogle Cloudの短期credentialを取得する。
3. commit SHAで識別できるimageをArtifact Registryへpushし、image digestを記録する。
4. 新Cloud Run revisionを既存trafficへ即時反映せずにdeployする。
5. 新revisionに対して `/health/live`、`/health/ready`、Catalog GET、最小simulation、GitHub Pages originのCORSを確認する。
6. 成功後にproduction trafficを新revisionへ移す。
7. Cloud Run service URLをGitHub Environmentの `VITE_API_BASE_URL`へ設定し、Pagesをbuild・deployする。
8. deployed PagesからCatalog取得と最小simulationを再確認する。

GitHubへGoogle Cloud service account JSON keyを保存しない。Workflowのprincipal条件はこのrepositoryとproduction environmentへ限定する。

### 実装（`#106` `M45-INFRA-002`）

- [`.github/workflows/pr.yml`](../.github/workflows/pr.yml)の`container` jobが、PRごとに
  `mise run test:container`（production container build + local Docker smoke test）を実行する
  （Cloud Runへはdeployしない）。
- [`.github/workflows/main.yml`](../.github/workflows/main.yml)の`deploy` jobが、`quality`
  job成功後だけ実行される。`google-github-actions/auth`でWorkload Identity Federationの
  短期credentialを取得し（`google-github-actions/setup-gcloud`でgcloud CLIへ引き継ぐ）、
  [`scripts/cloud-run/02-build-and-push-image.sh`](../scripts/cloud-run/02-build-and-push-image.sh)
  でimageをbuild・push、image digestを記録する。
- [`scripts/cloud-run/ci-deploy-candidate.sh`](../scripts/cloud-run/ci-deploy-candidate.sh)が、
  `gcloud run services replace`一回の呼び出しで、既存revisionへtraffic 100%を明示的に固定
  したまま、新revisionを`candidate` tag付き・traffic 0%で作成する（`gcloud run services
replace`は`--no-traffic`を持たないため、manifest自体の`spec.traffic`で宣言する。
  [`apps/api/src/infrastructure/deploy/cloud-run-manifest.ts`](../apps/api/src/infrastructure/deploy/cloud-run-manifest.ts)
  が該当ロジックをunit testで検証する）。traffic切替の隙間は生じない。「既存revision」は
  Serviceの`status.traffic`でpercentが厳密に100のrevisionだけを正とする
  （[`apps/api/src/infrastructure/deploy/cloud-run-traffic.ts`](../apps/api/src/infrastructure/deploy/cloud-run-traffic.ts)
  の`resolveCurrentRevisionName`）——`status.latestReadyRevisionName`は使わない。
  smokeに失敗して一度もpromoteされていないcandidateもReady状態にはなり得るため、それを
  「現在productionのrevision」と誤認して次回deployでtraffic 100%を割り当ててしまう
  （PRレビュー指摘 #112 P1、初回レビュー）。
- CIは初回Cloud Run deployを行わない——最初のrevisionは
  [`scripts/cloud-run/03-deploy-service.sh`](../scripts/cloud-run/03-deploy-service.sh)
  （一度限りの手動セットアップ）で事前に作成済みの前提とする。`gcloud run
services describe`の失敗（一時的なネットワーク障害等）やservice未作成により
  現在100% trafficを受けているrevisionを特定できない場合、それを「初回deploy
  （bootstrap）」として扱い新revisionへ即100% trafficを割り当てることは、Issue
  #106の安全条件「smoke test成功後にのみtrafficを確定する」に反するため、
  `ci-deploy-candidate.sh`・[`apps/api/src/infrastructure/deploy/render-cloud-run-manifest-cli.ts`](../apps/api/src/infrastructure/deploy/render-cloud-run-manifest-cli.ts)・`buildCandidateTrafficTargets`の3層でfail closedする
  （PRレビュー指摘 #112、2026-07-15、5回目）。
- rollback先は「直近のReady revision」では選ばない。複数回連続でcandidateがsmokeに
  失敗すると、未promoteの失敗revisionがReadyのまま残り、`candidate` tagは常に最新の
  失敗revisionへ移るため、tagだけの除外でも古い失敗revisionを再選択し得る
  （PRレビュー指摘 #112 P1、2026-07-15再レビュー）。代わりに、promote成功時にだけ
  更新される永続的な2つのtagで判定する——`stable`（現在100% trafficの、最後に
  promoteされたrevision）と`stable-previous`（直前にpromoteされていたrevision、
  1段階のrollback履歴）。[`scripts/cloud-run/ci-promote-traffic.sh`](../scripts/cloud-run/ci-promote-traffic.sh)
  がpromote成功時にこの2つのtagを回す——`--to-revisions`（traffic 100%切替）と
  `--update-tags`（tag rotation）は同一の`gcloud run services update-traffic`
  呼び出しにまとめる。2回の別呼び出しに分けると、traffic切替後にtag更新だけが
  失敗した場合、新revisionが既にproductionなのに`stable`が旧revisionを指した
  ままになり、以後のrollback先・次回promoteの履歴判定を誤る
  （PRレビュー指摘 #112 P2、2026-07-15 3回目レビュー）。
  `gcloud run services replace`はdeployのたびに`spec.traffic`全体を新しい
  desired stateとして適用するため、`ci-deploy-candidate.sh`は`stable`／
  `stable-previous`タグを毎回のmanifestへ明示的に再宣言し、deploy attempt
  （成功・失敗いずれでも）で失われないようにする
  （`buildCandidateTrafficTargets`のunit testが検証する）。
- candidateのCatalog GETから選択可能なUnitを探し、
  [`apps/api/src/infrastructure/deploy/simulation-smoke-request.ts`](../apps/api/src/infrastructure/deploy/simulation-smoke-request.ts)
  で最小simulation requestを組み立てる。選択可能なUnitが無い場合は`deploy` jobをここで
  失敗させる——simulation smoke testを黙ってskipしない（PRレビュー指摘 #112 P1-3）。
- [`scripts/cloud-run/04-smoke-test.sh`](../scripts/cloud-run/04-smoke-test.sh)を
  `candidate` tag付きURLに対して実行する。live・ready・Catalog GET・最小simulationに加え、
  CORS preflight（OPTIONS）だけでなく実際のcross-origin GET/POSTでも
  `Access-Control-Allow-Origin`を確認する。失敗した場合はjobがここで停止し、
  [`scripts/cloud-run/ci-promote-traffic.sh`](../scripts/cloud-run/ci-promote-traffic.sh)
  （traffic 100%への昇格）は実行されない——新revisionへproduction trafficは流れない。
- 成功後、`deploy` jobがCloud Run service URLを`production` GitHub Environmentの
  `VITE_API_BASE_URL` variableへ書き込む（`gh variable set`。既定の`GITHUB_TOKEN`では
  repository/environment variableを書き換えられないため、Variables書き込みのみに
  絞ったfine-grained PATを`VARS_ADMIN_TOKEN` secretとして使う）。Pagesのbuild・deploy
  自体は`#99`（`M45-UI-007`）のスコープ。
- Workload Identity Pool／Provider・最小権限deploy service accountは
  [`scripts/cloud-run/00-bootstrap-ci-cd.sh`](../scripts/cloud-run/00-bootstrap-ci-cd.sh)
  （project IAM adminが一度だけ手動実行）で作成する。OIDC providerの`attribute-condition`は
  このrepositoryと`production` GitHub Environmentへ限定する。
- Billing budget alertとlog保持期間は
  [`scripts/cloud-run/00b-configure-billing-and-logging.sh`](../scripts/cloud-run/00b-configure-billing-and-logging.sh)
  （Billing Account Administratorが一度だけ手動実行）で設定する。

## Cloud Run rollback

1. 障害がUI、Cloud Run到達性、application、Catalog revisionのどこにあるかをRequest IDとrevisionで切り分ける。
2. 新revisionのtrafficを0へ戻し、直前にsmoke test成功済みのrevisionへ100%移す
   （[`.github/workflows/rollback-cloud-run.yml`](../.github/workflows/rollback-cloud-run.yml)を
   `workflow_dispatch`で手動実行する。`target_revision`省略時は、Serviceの
   `stable-previous` tag（promote成功時にだけ更新される、直前にpromoteされていた
   revisionの永続的な記録）が指すrevisionを自動検出する
   [`scripts/cloud-run/ci-rollback-traffic.sh`](../scripts/cloud-run/ci-rollback-traffic.sh)を使う。
   「直近のReady revision」は使わない——未promoteの失敗revisionもReadyになり得るため
   （PRレビュー指摘 #112 P1、2026-07-15再レビュー）。`stable-previous` tagが無い
   （promote実績が1回以下、または既にrollback済み）場合はjobを失敗させるため、
   `target_revision`を明示する）。traffic切替・`stable`更新・`stable-previous`削除は
   同一の`gcloud run services update-traffic`呼び出しにまとめる。rollback後も
   `stable-previous`を残すと、次回の自動rollbackが「現在trafficを受けている
   revision（rollback直後のtarget自身）」を再び選び、no-opのまま成功扱いになって
   しまうため、rollback成功時に必ず`stable-previous`を削除する
   （PRレビュー指摘 #112 P1、2026-07-15 3回目レビュー）。より過去のrevisionへ
   さらにrollbackしたい場合は`target_revision`を明示する。
3. `/health/ready`、Catalog GET、最小simulationを確認する（rollback workflowが、
   main deployと同様にrollback先のCatalogから選択可能なUnitを探してrequestを
   組み立て、`04-smoke-test.sh`を自動で再実行する。選択可能なUnitが無い場合は
   rollback workflow自体を失敗させる——simulation smoke testを黙ってskipしない。
   PRレビュー指摘 #112 P1、2026-07-15再レビュー）。
4. API URL自体が変わらない場合はPagesを再deployしない。変更した場合だけGitHub Environmentを戻してPagesを再buildする。
5. 失敗したimage digest、Git SHA、Cloud Run revision、Catalog revision、diagnostic IDを障害記録へ残す
   （`deploy` jobの`GITHUB_STEP_SUMMARY`に記録済みの値を参照できる）。

maximum instancesは費用の絶対上限ではない。Google Cloud Billing budget alert、Cloud Run instance count、request count、5xx、latencyを確認し、想定外の利用時はservice停止またはinvocation方針の再検討を行う。

## 構造化ログ

`LOG_LEVEL`で有効化されるFastify内蔵のpinoロガーがJSON形式で出力する。

- 全リクエストで`requestId`フィールドが自動で付く（`requestIdLogLabel`設定と、
  `X-Request-Id`ヘッダーまたは生成UUIDを解決する`genReqId`による）。
- 戦闘完了時に`catalogRevision`・`battleId`・`outcome`・`completionReason`・
  `completedTurn`・`eventCount`・`stateTransitionCount`を1行で記録する。
- 500番台のエラーは`diagnosticId`をレスポンスとログの両方へ記録し、事後に
  サーバーログと照合できるようにする。

生のリクエスト・レスポンス本文やCatalog全体は記録しない
（`11_インフラストラクチャ設計.md`「ログ設計」参照）。
