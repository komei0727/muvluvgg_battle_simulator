# 戦闘シミュレーションUI設計

## 目的

本ディレクトリは、GitHub Pagesで公開する戦闘シミュレーションUIの設計上の正本である。
人間によるレビューだけでなく、AIが要求を推測せず実装・テストできる粒度を目指す。

採用済みの画面イメージは [battle-simulator-mock.html](./battle-simulator-mock.html) とする。
モック内の戦闘結果、API接続状態、Catalog版、ユニット画像は例示であり、実データ契約ではない。

## 正本の優先順位

矛盾がある場合は、次の優先順位で判断する。

1. HTTP契約は [../ddd/10_API設計.md](../ddd/10_API設計.md) とOpenAPIを正本とする。
2. UIの挙動とデータ変換は本ディレクトリのMarkdown文書を正本とする。
3. 色、余白、密度、全体配置は採用済みHTMLモックを視覚上の正本とする。
4. HTMLモック内の固定値と疑似ロジックは正本にしない。

APIの実装と設計書が食い違う場合、UI側で黙って吸収せずAPI契約の差異として扱う。

## 確定した設計判断

| ID           | 判断                                                                                                               |
| ------------ | ------------------------------------------------------------------------------------------------------------------ |
| `UI-ADR-001` | 1ページ内で条件設定、実行、サマリ、詳細確認を完結させる。                                                          |
| `UI-ADR-002` | 味方と敵はそれぞれ前衛3枠・後衛3枠として表示する。API送信時の後衛値は `REAR` とする。                              |
| `UI-ADR-003` | ユニットとメモリーは専用ダイアログから選択する。                                                                   |
| `UI-ADR-004` | サマリは `DAMAGE`、`DEFENSE（被ダメージ）`、`HEAL` を常設する。未対応の回復は0表示する。                           |
| `UI-ADR-005` | 詳細は時系列イベント、状態遷移、レスポンスJSONの3表示とする。                                                      |
| `UI-ADR-006` | ゲーム画面を模倣せず、データ分析基盤としてシステマティックな外観にする。                                           |
| `UI-ADR-007` | 画像は任意とし、未設定時のフォールバック表示を必須にする。                                                         |
| `UI-ADR-008` | `GET /api/v1/battle-simulation-catalog` をUnit・Memory一覧と選択可否の正本とする。静的Catalogを同梱しない。        |
| `UI-ADR-009` | 未実装Capabilityを必要とする定義はAPIが一覧へ含めて `selectable: false` とする。UIは理由を表示し、選択不可にする。 |
| `UI-ADR-010` | GitHub PagesとAPIは別オリジンとし、API側で許可オリジンを限定したCORSを設定する。                                   |
| `UI-ADR-011` | APIのM4.5配備先にGoogle Cloud Runを採用し、GitHub PagesはCloud Runの公開HTTPS URLをbuild時設定として使用する。     |

## 変更管理

- 挙動を変更する場合は、先に該当Markdownと受け入れ条件を更新する。
- 視覚変更はHTMLモックにも反映する。
- API DTO変更は `03_API・データ連携設計.md` とAPI設計書を同じPRで更新する。
- M5以降の追加イベントは、未知イベントを壊さない既存方針を保ちながら表示辞書を追加する。
- 未決事項を実装者が暗黙に決めず、`未決事項`として文書へ追加する。

## 現時点の前提

- バックエンドはM4完了時点で、`POST /api/v1/battle-simulations` を同期実行する。
- Unit・Memory一覧APIとAPIのCORSをM4.5で実装し、UI実装の前提とする。
- APIはM4.5でGoogle Cloud Runの `asia-northeast1` へ配備し、scale-to-zeroを許容する。
- UIは一覧APIの取得に成功するまで編成編集を開始させない。
- 認証、結果保存、共有URL、複数回比較は初期スコープ外である。
