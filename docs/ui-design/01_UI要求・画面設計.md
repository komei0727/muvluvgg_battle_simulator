# UI要求・画面設計

## 1. 目的

利用者が戦闘条件を設定し、APIへ1回のシミュレーションを依頼し、結果をサマリと詳細データの両方から検証できるUIを提供する。

本UIはゲームプレイ画面ではなく、戦闘ルールとCatalogを検証するデータ分析基盤である。

## 2. 対象利用者

- 戦闘ルールを設計・検証する開発者
- Catalog定義の挙動を確認する担当者
- 実装マイルストーンの受け入れ確認を行うレビュアー

初期版では一般ゲーム利用者向けの説明や演出より、入力と結果の追跡可能性を優先する。

## 3. ユースケース

### `UI-UC-001` 戦闘条件を設定する

1. UIは一覧APIからUnit・Memoryと選択可否を取得する。
2. 利用者は味方と敵のユニットを各陣営1～5体選択する。
3. 各ユニットを前衛または後衛の空き枠へ配置する。
4. 利用可能なメモリーを各陣営0～6件選択する。
5. ターン上限1～99とログレベルを設定する。
6. UIは入力違反を戦闘開始前に表示する。

画面には各陣営6枠を表示するが、APIが受理するユニット数は最大5体である。6枠目まで埋めようとした場合は選択を確定せず、「1陣営に設定できるユニットは5体までです」と表示する。

### `UI-UC-002` 戦闘を実行する

1. 利用者が「戦闘を開始」を押す。
2. UIは入力を再検証し、違反がなければAPI DTOへ変換する。
3. 実行中は重複送信を防ぎ、進行中表示とキャンセル操作を提示する。
4. 成功時は戦闘サマリと詳細データを表示する。
5. 失敗時は入力違反、混雑、タイムアウト、サーバー障害を区別して表示する。

### `UI-UC-003` 戦闘サマリを確認する

成功結果から次を表示する。

- 勝敗
- 終了理由
- 終了ターン / 設定ターン上限
- Battle ID
- Catalog revision
- 各ユニットの与ダメージ、被ダメージ、回復、生存状態

### `UI-UC-004` 詳細データを確認する

利用者は次のタブを切り替えられる。

- 時系列イベント
- 状態遷移
- レスポンスJSON

イベントは機械コードだけでなく日本語の要約を表示する。未知イベントは除外せず、汎用表示へフォールバックする。

### `UI-UC-005` 条件を変更して再実行する

- 「条件を編集」で条件設定領域へ戻れる。
- 前回の条件と結果は、再実行を開始するまで保持する。
- 「再実行」は現在の条件を再送する。乱数を含むため同じ結果になるとは限らない。
- 新しい実行が成功した時点で前回結果を置換する。
- 新しい実行が失敗した場合、前回成功結果は残し、その上にエラーを表示する。

## 4. ページ構成

```text
AppShell
├── Header
│   ├── Product identity
│   └── API/Catalog status
├── PageHeading
├── BattleSetupSection
│   ├── AllyFormationEditor
│   │   ├── Unit slots: FRONT 3 / REAR 3
│   │   └── Memory slots: 6
│   ├── VS divider
│   ├── EnemyFormationEditor
│   │   ├── Unit slots: FRONT 3 / REAR 3
│   │   └── Memory slots: 6
│   └── ExecutionParameters
├── SubmissionFeedback
├── BattleSummarySection
└── BattleDetailsSection
    ├── Event timeline
    ├── State transitions
    └── Raw JSON
```

結果領域は初期表示では存在してもよいが非表示とする。成功後、設定領域の下に表示し、サマリ先頭へスクロールする。設定領域を自動的に折りたたまない。

## 5. 条件設定領域

### 5.1 編成枠

各陣営は以下の順で表示する。

```text
FRONT / 前衛: column 0 | column 1 | column 2
REAR  / 後衛: column 0 | column 1 | column 2
```

- 空き枠は追加操作であることを `＋` とラベルで示す。
- 選択済み枠は画像またはフォールバック、表示名、属性、ロールを表示する。
- 枠を押すとユニット選択ダイアログを開く。
- 選択済み枠では変更と解除ができる。
- 同じユニット定義を複数枠へ配置できる。
- 同一陣営の同じ座標へ複数配置できない構造にする。
- 適性外の前後列にも配置できる。サーバーが適性補正を適用するため、UIは警告を表示するが禁止しない。

### 5.2 ユニット選択ダイアログ

必須機能：

- 表示名・定義IDの部分一致検索
- 属性による絞り込み
- ロールによる絞り込み
- 前後列適性の表示
- 現在選択中のユニットの識別
- 枠を空にする操作
- 未対応Capabilityによる利用不可表示

選択不可項目は一覧から消さない。無効状態とし、必要なCapability IDを表示する。これによりマイルストーン進行による解禁対象を把握できる。

Unit一覧、表示情報、選択可否、未対応Capabilityは `GET /api/v1/battle-simulation-catalog` の値を使用する。UIがCatalogファイルやIDから可否を再計算しない。

### 5.3 メモリー

- 各陣営に6枠表示する。
- 空き枠を押すとメモリー選択ダイアログを開く。
- 表示名・定義IDで検索できる。
- 同じメモリーIDの重複はAPI契約上許容されるため、UIでも禁止しない。
- 未実装Capabilityを必要とするメモリーは選択不可とし、理由を表示する。
- M4時点のproduction Catalogでは全メモリーが未実装Capabilityを要求するため、初期実装では全件が選択不可となり得る。
- Memory一覧、表示情報、選択可否、未対応Capabilityは一覧APIの値を使用する。

### 5.4 実行パラメータ

| 項目         | UI               | 既定値     | 制約                                |
| ------------ | ---------------- | ---------- | ----------------------------------- |
| ターン上限   | number input     | `10`       | integer、1～99                      |
| ログレベル   | select           | `DETAILED` | `SUMMARY`、`DETAILED`、`DIAGNOSTIC` |
| API endpoint | 読み取り専用表示 | ビルド設定 | 利用者が画面上で変更しない          |

`DIAGNOSTIC`にはレスポンス肥大の説明を付ける。`SUMMARY`でも状態遷移は全件返ることを前提とする。

### 5.5 戦闘開始ボタン

次を満たすと有効にする。

- 味方が1～5体
- 敵が1～5体
- 選択定義にUIが把握している未対応Capabilityがない
- ターン上限が1～99の整数
- ログレベルが許容値
- 実行中でない

無効時はボタンだけで理由を表現せず、設定領域上部に不足条件を表示する。

## 6. 実行状態

| 状態       | 表示                                                       |
| ---------- | ---------------------------------------------------------- |
| 未実行     | 結果領域を非表示。「DRAFT / NOT EXECUTED」                 |
| 入力エラー | 対象フィールドと設定領域上部にエラー                       |
| 実行中     | ボタンを無効化し「実行中…」。キャンセルボタンを表示        |
| 成功       | サマリ・詳細を表示し、完了時刻を表示                       |
| 失敗       | エラー概要、再試行可否、Request IDまたはdiagnosticIdを表示 |
| キャンセル | 「実行をキャンセルしました」。前回成功結果は保持           |

実行開始から800ms未満で完了した場合も、不要な点滅を避けるため大きなローディングオーバーレイは使用しない。ボタン内進行表示は即時反映する。

一覧APIの初期取得中は編成領域にskeletonを表示する。取得失敗時は編成・戦闘開始を無効にし、ネットワーク/CORS候補を示すエラーと手動再読込を表示する。古いbundle内Catalogへのfallbackは行わない。

## 7. 戦闘サマリ

### 7.1 結果ヘッダー

| API値                 | 表示例                 |
| --------------------- | ---------------------- |
| `ALLY_WIN`            | `ALLY WIN / 味方勝利`  |
| `ALLY_LOSE`           | `ALLY LOSE / 味方敗北` |
| `ENEMY_DEFEATED`      | `敵陣営全滅`           |
| `ALLY_DEFEATED`       | `味方陣営全滅`         |
| `SIMULTANEOUS_DEFEAT` | `同時全滅`             |
| `TURN_LIMIT_REACHED`  | `ターン上限到達`       |

未知の列挙値はコードをそのまま表示し、画面全体を失敗させない。

### 7.2 ユニット別表

味方・敵を左右に並べ、次の列を常設する。

| 列                     | 意味               | 初期集計                                               |
| ---------------------- | ------------------ | ------------------------------------------------------ |
| `DAMAGE`               | 与えた実HPダメージ | `DAMAGE_APPLIED.details.hitPointDamage` を攻撃元で合計 |
| `DEFENSE / 被ダメージ` | 受けた実HPダメージ | 同値を対象で合計                                       |
| `HEAL`                 | 実HP回復量         | M7以前は0。回復イベント契約追加後に集計                |
| `STATUS`               | 最終状態           | `finalState.units[].combatStatus`                      |

オーバーキルを含む計算ダメージではなく、実際にHPを減らした `hitPointDamage` を使用する。

### 7.3 表示順

`initialState.units`の順序を維持する。定義IDではなく `battleUnitId` を同一性に使用し、同じ定義を複数配置した場合も別行として表示する。

## 8. 戦闘詳細

### 8.1 時系列イベント

- `events`の配列順ではなく `sequence` 昇順で表示する。
- 欠番を許容する。
- 表示項目は sequence、turn/cycle、type、source、targets、要約、state versionとする。
- イベントを展開すると `details` と因果参照を表示する。
- `stateTransitionIndex`がある場合、対応状態遷移へ移動できる。
- SUMMARYで省略された親イベントの参照があってもエラーにしない。

### 8.2 状態遷移

- `stateTransitions`の配列順で表示する。
- version before/after、causedBySequence、対象、deltaを表示する。
- deltaはツリー表示とJSON表示を切り替えられる。
- 全差分のクライアント側復元は初期必須機能にしないが、テスト用の純粋関数として検証可能にする。

### 8.3 レスポンスJSON

- APIレスポンスを改変せず整形表示する。
- コピー操作を用意する。
- 大きなレスポンスでもDOMノードをフィールドごとに大量生成しない。`pre`または遅延構築されたJSON viewerを使う。
- JSON内のIDや数値を翻訳しない。

## 9. 画像の扱い

- `imageUrl`はUI向けCatalog DTOの任意項目とする。
- ユニット画像とメモリー画像は同じ画像コンポーネントを使わず、用途別アスペクト比を許容する。
- URLなし、ロード失敗、空文字列ではフォールバックを表示する。
- フォールバックは表示名から生成した2文字、属性色、種別ラベルで構成する。
- 画像が操作名の唯一の情報になってはならない。
- 外部画像ホストを使用する場合は利用権、CSP、可用性を別途確認する。初期版は同一GitHub Pages成果物内の画像を推奨する。

## 10. 初期スコープ外

- 認証、ユーザー別設定
- 編成のサーバー保存
- 戦闘結果のサーバー保存・共有
- 複数結果の比較、統計解析
- 乱数seed指定
- 戦闘アニメーション
- Catalog編集
- API URLの利用者入力

## 11. 画面受け入れ条件

- `UI-AC-001`: 1ページ内で設定、実行、サマリ、詳細確認が完結する。
- `UI-AC-002`: 敵味方にFRONT 3枠・REAR 3枠が表示される。
- `UI-AC-003`: ユニット枠から検索・絞り込み可能な選択ダイアログを開ける。
- `UI-AC-004`: 各陣営6個のメモリー枠を操作できる。
- `UI-AC-005`: 1～5体の制約とターン1～99を送信前に検証する。
- `UI-AC-006`: 実行中に同じ操作から重複送信しない。
- `UI-AC-007`: 成功結果に勝敗、理由、ターン、Battle ID、Catalog revisionを表示する。
- `UI-AC-008`: 同じunitDefinitionIdの参加者をbattleUnitId単位で別行表示する。
- `UI-AC-009`: DAMAGE、被ダメージ、HEAL、STATUSを敵味方別に表示する。
- `UI-AC-010`: イベント、状態遷移、JSONを切り替えられる。
- `UI-AC-011`: 未知イベントと未知列挙値でページ全体がクラッシュしない。
- `UI-AC-012`: APIエラーのcode、違反箇所、追跡IDを表示できる。
- `UI-AC-013`: 画像なし・画像ロード失敗でも選択と結果確認ができる。
- `UI-AC-014`: 未対応Capabilityを必要とする定義を選択不可として理由表示する。
- `UI-AC-015`: スマートフォン幅でも横方向の前衛3枠・後衛3枠という意味が失われない。
- `UI-AC-016`: 一覧APIの取得に成功するまで編成を編集できず、失敗時に手動再読込できる。
- `UI-AC-017`: Unit・Memoryの表示と選択可否を一覧APIだけから決定する。
