# UIテスト戦略

## 1. 目的

UIの見た目だけでなく、API DTO変換、battleUnitId単位の集計、未知イベントへの耐性、GitHub Pages配備を継続的に検証する。

テストは実装詳細ではなく、[01_UI要求・画面設計.md](./01_UI要求・画面設計.md) の受け入れ条件と本書のTest IDへ紐づける。

## 2. テストレベル

| レベル            | 対象                                                            | 実行頻度              |
| ----------------- | --------------------------------------------------------------- | --------------------- |
| Pure unit         | validator、request mapper、summary projector、formatter、filter | PRごと                |
| Component         | slot、dialog、form、summary、tabs、error表示                    | PRごと                |
| Contract fixture  | 実OpenAPI/response fixtureとの互換                              | PRごと                |
| E2E mock API      | browser上の主要フロー                                           | PRごと                |
| E2E deployed API  | Pages相当originからCORSを含む縦切り                             | mainまたは手動        |
| Visual regression | 採用モックとの主要viewport比較                                  | UI変更PR              |
| Accessibility     | 自動検査＋keyboard/screen reader手動                            | PR自動＋release前手動 |

## 5. Contract fixtures

fixtureを次に分ける。

```text
ui/src/test/fixtures/
├── m4.5-catalog.json
├── m4.5-catalog-unavailable.json
├── m4-success-minimal.json
├── m4-success-duplicate-definition.json
├── success-unknown-event.json
├── error-invalid-command.json
├── error-unsupported-rule.json
├── error-capacity.json
└── malformed-success.json
```

- 手作業でAPIの巨大responseを再現せず、backend contract testまたは実OpenAPI fixtureから更新する。
- fixtureへ日時やrandom IDの不要な変動を入れない。
- backend wire DTOに変更があったPRはUI fixture testも実行する。
- OpenAPI型生成を導入した場合、生成差分がcleanであることをCIで確認する。

## 7. Visual regression

採用HTMLモックから次の視覚的不変条件を抽出する。

- dark systematic theme
- 味方blue、敵pink
- desktopで左右編成
- FRONT/REAR各3枠
- parameter barと主要submit CTA
- outcome strip
- 味方・敵summary table
- 3つのdetails tab

1440×900と390×844を最低baselineとする。動的時刻、Battle ID、画像取得を固定して撮影する。

pixel完全一致だけを合否にせず、意図した変更ではbaselineをレビュー更新する。HTMLモックをPlaywright testの実装依存として読み込まない。

## 9. CI品質ゲート

PRで必須：

1. format check
2. typecheck
3. lint
4. unit/component test
5. UI build
6. E2E mock API
7. accessibility自動検査

mainで追加：

1. backend既存full quality gate
2. Pages artifact build
3. deploy
4. deployment smoke test

Live API E2EはAPI環境の可用性に依存するため、初期は別jobとし、必須化の可否を運用実績から決める。失敗を無視するのではなくjob statusを残す。

## 10. Coverage方針

- line coverageの数値達成だけを目的にしない。
- request mapper、validator、summary projector、error normalizerはbranch coverage 100%を目標とする。
- componentは主要利用者操作とfailure stateを網羅する。
- CSSとAPI contract fixtureをcoverage対象から除外してよい。
- snapshot testだけでcomponent挙動を保証しない。
