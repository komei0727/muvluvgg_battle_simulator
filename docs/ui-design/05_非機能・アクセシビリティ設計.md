# 非機能・アクセシビリティ設計

## 1. 対象

採用済みHTMLモックのシステマティックな外観を維持しながら、GitHub Pages上で入力・結果確認を安全に行うための品質基準を定義する。

アクセシビリティ目標はWCAG 2.2 AA相当とする。自動検査だけで適合を宣言せず、キーボードと読み上げの手動確認を完了条件に含める。

## 2. ビジュアル原則

- データ分析基盤として、情報階層、比較、追跡可能性を優先する。
- 青を味方、ピンクを敵に使用するが、色だけで陣営を区別しない。
- 暗い背景、高コントラストの本文、低彩度の罫線を基本とする。
- 装飾画像やゲーム固有UIを模倣しない。
- アニメーションは状態変化を補助する範囲に限定する。

## 3. Design tokens

採用モックを初期tokenの正本とし、CSS custom propertiesへ集約する。

```css
:root {
  color-scheme: dark;
  --color-bg: #07101d;
  --color-surface: #0d1929;
  --color-surface-raised: #111f31;
  --color-surface-strong: #17273a;
  --color-border: #263a50;
  --color-border-strong: #36536d;
  --color-text: #edf6ff;
  --color-text-muted: #8fa6ba;
  --color-ally: #45c8ff;
  --color-enemy: #ff6b9a;
  --color-success: #55e6ae;
  --color-warning: #ffc861;
  --color-danger: #ff7285;
  --radius-sm: 6px;
  --radius-md: 10px;
}
```

### Typography

- UI本文はOS標準の日本語sans-serifを優先し、外部font取得を必須にしない。
- ID、event type、数値、statusは等幅font stackを使う。
- 本文の最小基準は12pxとし、補助ラベルの9～10pxには十分なcontrastとletter spacingを確保する。
- 重要な数値は桁区切りするが、JSON表示内の値は変換しない。

### Spacing

4px基準のspacing scaleを使用する。

```text
4 / 8 / 12 / 16 / 20 / 24 / 32 / 40 / 48
```

新規componentで任意の細かな値を増やさず、例外には理由をコメントする。

## 4. レスポンシブ設計

### `>= 1041px`

- 味方、VS、敵を横3列に配置する。
- サマリの味方・敵を左右に配置する。
- 実行parameterを1行に配置する。

### `761px–1040px`

- 味方、VS、敵を縦に配置する。
- 各陣営内のFRONT 3枠・REAR 3枠は横3列のまま保つ。
- parameterを2列へ折り返す。
- サマリは情報が読める幅なら左右、狑い場合は縦へ切り替える。

### `<= 760px`

- page左右marginを12pxとする。
- 味方・敵、サマリを縦積みにする。
- 編成の3column関係は維持する。2列へ組み替えない。
- unit cardは表示名を省略可能だが、accessible nameには完全名を残す。
- memory 6枠は横6列を基本とし、最小tap targetを満たせない場合は3列×2段を許容する。その場合もslot indexを表示する。
- summaryは横scrollまたは優先列表示を使う。`DAMAGE`、`DEFENSE`、`STATUS`を非表示にしない。`HEAL`は横scroll先でもよい。
- 詳細tabは横scroll可能にし、ラベルを省略しない。

### 検証viewport

- 1440 × 900
- 1024 × 768
- 768 × 1024
- 390 × 844
- 320 × 568

横スクロールはsummary table、JSON、詳細tableの内部に限定し、page全体には発生させない。

## 5. キーボード操作

| 対象             | 操作                                                                 |
| ---------------- | -------------------------------------------------------------------- |
| Unit/Memory slot | Tabで移動、Enter/Spaceでdialogを開く                                 |
| Dialog           | 開いたら検索へfocus、Tabをdialog内に保持、Escapeで閉じる             |
| Catalog item     | Tabで選択buttonへ移動、Enter/Spaceで確定                             |
| Tabs             | Arrow Left/Rightで移動、Home/Endをサポート、選択tabだけ `tabIndex=0` |
| Submit           | Enterで意図しない送信が起きない。明示button操作                      |
| Event row        | Enter/Spaceで展開・折りたたみ                                        |
| JSON copy        | buttonとして操作可能                                                 |

Dialogを閉じたら起点slotへfocusを戻す。実行成功時はサマリ見出しへプログラムfocusし、`tabIndex=-1`を使用する。

## 6. Semantic HTML / ARIA

- ページ領域に `header`、`main`、`section`、見出し階層を使用する。
- slotはclickable `div`ではなく `button`とする。
- dialogはnative `<dialog>`を使える場合も、focus trapとbrowser互換をテストする。代替実装では `role="dialog"`、`aria-modal="true"`、labelを必須とする。
- filter入力には視覚ラベルまたはaccessible labelを付ける。
- tablist/tab/tabpanelの関連を `aria-controls` / `aria-labelledby`で設定する。
- 実行状態とエラー概要は `aria-live="polite"`へ通知する。
- 緊急でないAPI失敗に `role="alert"`を乱用しない。
- summary tableは本物の `table`、`thead`、`tbody`、scope付きheaderを使用する。
- 色付きbadgeにはテキストラベルを併記する。

## 7. Contrastと状態表現

- 通常本文と背景は4.5:1以上、大きな文字・UI境界は3:1以上を目標とする。
- focus indicatorをoutline削除だけで消さない。
- ally/enemy、success/error、selectable/unavailableを色だけで表現しない。
- disabled itemはopacityだけでなく「未対応」テキストとCapability理由を表示する。
- field errorは入力枠、メッセージ、summaryの3箇所を適切に関連づける。

## 8. Motion

- 結果表示のfade/translateは300ms以下とする。
- `prefers-reduced-motion: reduce`ではsmooth scrollと装飾transitionを無効化する。
- loading spinnerだけに依存せずテキストを併記する。
- 点滅、連続pulse、戦闘演出を追加しない。

## 9. 性能

初期目標：

| 項目              | 目標                                         |
| ----------------- | -------------------------------------------- |
| Initial JS        | gzip 250 KiB以下（画像を除く）               |
| CSS               | gzip 50 KiB以下                              |
| LCP               | 一般的な4G/中性能端末で2.5秒以下を目標       |
| Catalog filtering | 100件で入力から次frame以内                   |
| Interaction       | submit、tab切替、dialog openに目立つ遅延なし |

- Catalog全体のskill/effect定義をbundleしない。
- 画像へwidth/heightまたはaspect-ratioを指定しlayout shiftを防ぐ。
- 初期表示に不要なraw JSON viewerの高機能libraryを遅延loadしてよい。
- event件数が多い場合は段階描画するが、データ自体を破棄しない。
- responseをlocalStorageへ保存しない。

## 10. Browser compatibility

対象はGitHub Pagesと標準的なFetch/AbortController/dialogを利用できるevergreen browserとする。

- Chrome/Edge currentと1世代前
- Firefox currentと1世代前
- Safari currentと1世代前
- iOS Safari currentと1世代前

native dialogに対象browser上の問題がある場合は、小さなaccessibility対応dialog componentへ置換する。browser判定による分岐よりfeature testを優先する。

## 11. セキュリティ

### XSS

- Catalog displayName、API message、event detailsをHTML文字列として挿入しない。
- React text interpolationまたは `textContent`を使用する。
- `dangerouslySetInnerHTML`を禁止する。
- JSON syntax highlight導入時もHTML escapeを保証するlibraryだけを使う。

### Configuration

- GitHub Pages成果物にはsecretを置かない。
- API base URL、Pages base path、build revisionは公開情報とする。
- production API base URLはCloud Run service URLとし、GitHub EnvironmentからVite buildへ渡す。
- 利用者がAPI URLを入力できないため、任意送信先へのPOST機能にしない。

### CSP推奨値

GitHub Pagesでresponse headerを自由に設定できないため、可能な範囲でHTML meta CSPを使用する。Vite buildでinline script/styleを生成しない構成を前提に、API hostと画像hostを明示する。

```text
default-src 'self';
script-src 'self';
style-src 'self';
img-src 'self' data:;
connect-src 'self' https://<cloud-run-service-host>;
object-src 'none';
base-uri 'self';
form-action 'self';
```

外部画像hostを追加する場合は `img-src`だけを限定追加する。CSPを動的文字列連結で生成しない。

### Privacy

- 初期版は第三者analyticsを導入しない。
- battle request/responseを第三者へ送信しない。
- consoleへresponse全体、diagnostic data、Catalog定義全体を出力しない。
- Clipboard copyは利用者の明示操作時だけ行う。

## 12. 可用性と障害時UX

- API statusは初期表示で必須health requestにしない。health API失敗だけでsubmitを永久disableしない。
- submit結果を最終的な到達性判定とする。
- API混雑・timeout・network/CORSを区別可能な範囲で表示する。
- frontend Catalog読込に失敗した場合は条件設定を開始させず、再読込を案内する。
- Cloud Runのscale-to-zeroによる初回cold startをnetwork failureと断定せず、待機中表示と手動再試行を提供する。
- 画像失敗は局所fallbackとし、ページ障害にしない。
- GitHub Pagesのoffline cache/PWAは初期スコープ外とする。

## 13. 観測性

UI内に次の相関情報を表示できるようにする。

- UI build revision
- UI Catalog revision
- API response Catalog revision
- `X-Request-Id`
- `battleId`
- 500系の `diagnosticId`

通常画面では必要最小限を表示し、詳細またはエラー領域でコピーできるようにする。独自telemetryは初期導入しない。

## 14. 非機能受け入れ条件

- `UI-NFR-001`: mouseなしで条件設定、dialog選択、実行、結果tab確認ができる。
- `UI-NFR-002`: 390px幅でpage全体の横scrollが発生しない。
- `UI-NFR-003`: 320px幅でもFRONT/REARとcolumnの意味を判別できる。
- `UI-NFR-004`: 色覚に依存せず味方・敵・エラー・未対応を判別できる。
- `UI-NFR-005`: reduced motion設定でsmooth scrollと装飾animationを抑制する。
- `UI-NFR-006`: Catalog/API文字列がHTMLとして解釈されない。
- `UI-NFR-007`: build成果物にsecretやlocalhost API URLが含まれない。
- `UI-NFR-008`: 画像404でも操作と結果表示を継続できる。
- `UI-NFR-009`: Cloud Run cold start中も二重送信せず、timeout後に手動再試行できる。
