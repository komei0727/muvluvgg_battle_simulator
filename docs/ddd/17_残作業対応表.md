# 残作業対応表

## 目的

M7以降のRule、production Catalogの不完全変換、未変換Memory、実装Taskを一つの基準で追跡する。

機械可読な正本は [`17_残作業対応表.json`](./17_残作業対応表.json) とする。本書、[`13_実装計画.md`](./13_実装計画.md)、[`15_Unit_Memory変換台帳.md`](./15_Unit_Memory変換台帳.md) の件数を手動で個別管理しない。JSONと既存のRule Coverage・変換台帳・Catalog inventory・Capability registryの一致は `remaining-work.test.ts` で検証する。

## Taskへの割当

JSONは次の対応を保持する。

- `tasks`: Task ID、GitHub Issue番号、Phase、Milestone
- `ruleAssignments`: 未完了Rule IDを完了責任を持つTaskへ一意に割り当てる
- `conversionThemeAssignments`: 不完全変換テーマと行数を解消責任を持つTaskへ割り当てる
- `unconvertedMemoryAssignments`: 未変換Memory 26件をCatalog変換Taskへ一意に割り当てる

一つのRuleを複数Taskでテストしてよいが、完了責任を持つTaskは`ruleAssignments`上で一つだけとする。不完全変換行はproduction Catalog統合テストが成功するまで台帳から除去しない。

## 更新手順

1. 新しい残作業を発見したら、先に該当するRule、変換台帳行、Memory、Capabilityを特定する。
2. 既存Taskで完了できない場合だけRoadmap Issue #158配下へTaskを追加する。
3. JSONのTask割当と、対応する設計書・変換台帳を同じPRで更新する。
4. `remaining-work.test.ts`を含むquality gateを成功させる。
5. Ruleまたは変換行の完了は、production経路の受け入れ条件を満たしたPRでだけ計上する。

SchemaやMapperが定義を受理するだけでは完了としない。実ライフサイクル、Domain Event、StateDelta復元、production Catalog統合テストまでを完了境界とする。
