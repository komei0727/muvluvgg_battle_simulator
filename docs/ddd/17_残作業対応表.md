# 残作業対応表

## 目的

M7以降のRule、production Catalogの不完全変換、未変換Memory、実装Taskを一つの基準で追跡する。

機械可読な正本は [`17_残作業対応表.json`](./17_残作業対応表.json) とする。本書、[`13_実装計画.md`](./13_実装計画.md)、[`15_Unit_Memory変換台帳.md`](./15_Unit_Memory変換台帳.md) の件数を手動で個別管理しない。JSONの`current`と既存のRule Coverage・変換台帳・Catalog inventory・Capability registryの一致は `remaining-work.test.ts` で検証する。

## 2026-07-20基準

| --------------- | ---: | -------------------: | ------------------------: |
| Domain Rule     |  109 | 47（M2、M3、M5、M6） |      62（M7: 41、M8: 21） |
| production Unit |   69 |                   69 | 0（ただし不完全変換96行） |

Unitの不完全変換96行はM7相当62行、M8相当34行である。`UNIT_CI_SMOKE_TEST`はproduction Unit 69件には含めず、Catalog inventory上の合成Unit 1件として別計上する。

JSONの`baseline`はこの時点を保存する不変スナップショットであり、以後の進捗では変更しない。進捗に伴う件数は`current`だけを更新する。`remaining-work.test.ts`は初期件数をテストコードへ重複定義せず、`current`を実データから導出した件数と照合する。

## Taskへの割当

JSONは次の対応を保持する。

- `tasks`: Task ID、GitHub Issue番号、Phase、Milestone
- `current`: Rule、Catalog、Memory、Capabilityの現在件数
- `ruleAssignments`: 未完了Rule IDを完了責任を持つTaskへ一意に割り当てる
- `conversionThemeAssignments`: 不完全変換テーマと行数を解消責任を持つTaskへ割り当てる
- `unconvertedMemoryAssignments`: 現在の未変換MemoryをCatalog変換Taskへ一意に割り当てる

一つのRuleを複数Taskでテストしてよいが、完了責任を持つTaskは`ruleAssignments`上で一つだけとする。完了したRule、変換行、Memoryは対応するassignmentから除去する。`tasks`は完了後も履歴として残し、新しいTaskが必要な場合は追記する。不完全変換行はproduction Catalog統合テストが成功するまで台帳から除去しない。

## 更新手順

1. 新しい残作業を発見したら、先に該当するRule、変換台帳行、Memory、Capabilityを特定する。
2. 既存Taskで完了できない場合だけRoadmap Issue #158配下へTaskを追加する。
3. JSONの`current`・Task割当と、対応する設計書・変換台帳を同じPRで更新する。`baseline`は変更しない。
4. `remaining-work.test.ts`を含むquality gateを成功させる。
5. Ruleまたは変換行の完了は、production経路の受け入れ条件を満たしたPRでだけ計上する。

SchemaやMapperが定義を受理するだけでは完了としない。実ライフサイクル、Domain Event、StateDelta復元、production Catalog統合テストまでを完了境界とする。

Issue #166でCapabilityを後続Taskがproduction検証できる単位へ再分割したため、`current`は32件となった。M6でproduction検証済みだったSkillRuntime counterを`CAP_SKILL_RUNTIME_COUNTER`として明示し、`CAP_COOLDOWN_MANIPULATION`と合わせて`IMPLEMENTED`は2件である。Issue #166自身はEngine機能を追加せず、`baseline`の30件／1件は履歴として維持する。
