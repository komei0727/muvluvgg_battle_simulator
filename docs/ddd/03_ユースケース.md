# ユースケース

## 目的とスコープ

本書は、戦闘シミュレーションAPIを利用するクライアントから見たシステムの振る舞いを定義する。

システムは戦闘開始後のプレイヤー操作、ユニット・スキルの登録や更新、戦闘の中断・再開を扱わない。

## アクター

| アクター        | 役割                                                                                   |
| --------------- | -------------------------------------------------------------------------------------- |
| APIクライアント | 味方編成、敵編成、規定ターン数を指定して戦闘シミュレーションを要求し、結果を受け取る。 |

## システムが保持する情報

- ユニット定義
- ユニットが持つAS、PS、EXスキルの定義と定義順
- メモリー定義
- 属性相性などの共通戦闘ルール

これらを登録・変更するユースケースは本システムの対象外とする。

## UC-01 戦闘をシミュレーションする

### 目的

指定された編成同士の戦闘を、開始から勝敗確定まで自動的に実行する。

### 入力

- 味方編成
  - 1～5体のユニットID
  - 各ユニットの配置位置
- 敵編成
  - 1～5体のユニットID
  - 各ユニットの配置位置
- 各編成のメモリーID：最大6件
- 規定ターン数

### 事前条件

- 指定されたユニットIDがアプリケーション内に存在する。
- 各陣営のユニット数が1体以上5体以下である。
- 同じ陣営内で同一ユニットIDを複数指定でき、それぞれを別の参加枠として扱う。
- 配置位置が横3列、縦2列の範囲内である。
- 同じ陣営内で配置位置が重複していない。
- 各編成のメモリー指定が6件以下で、指定されたメモリーIDが存在する。
- 規定ターン数が1以上99以下である。

### 基本フロー

1. システムはリクエストの形式と編成を検証する。
2. システムはユニットIDに対応するユニット・スキル定義を読み出す。
3. システムは配置適性、編成ボーナスおよびメモリー効果を反映し、両陣営の戦闘開始状態を生成する。
4. システムは戦闘開始イベントと開始状態を記録する。
5. システムはターンを開始し、各ユニットのAPとPPを全回復する。ターン開始を表すドメインイベントを発行し、そのタイミングに対応するPSを判定する。
6. システムは行動可能なユニットを行動速度順に並べ、行動順キューを生成する。同速の場合は味方陣営、敵陣営の順、同陣営内では前列、絶対的な左列の順とする。同時に、EXゲージが満タンならEX、満タンでなければASとして、各ユニットの行動種別を予約する。
7. システムは行動順キューの先頭からユニットを取り出す。
8. 行動機会を得たユニットについて、キューに予約されたEXスキルまたはASを使用する。使用できない場合は待機する。
9. ASを選ぶ場合、定義順に使用可否を判定する。AP不足、対象不在などにより発動できないASを飛ばし、最初に使用可能なASを使用する。すべて発動不能なら待機する。
10. システムはスキル使用前後や効果適用など、処理の進行に応じたドメインイベントを発行する。各イベントで、そのタイミングに対応するPSの発動条件を判定する。
11. 同時に条件を満たしたPSでは、先制攻撃を持つ候補を通常候補より先に処理する。それぞれの候補群を所有者の行動速度順に処理し、同速時は味方陣営、敵陣営の順、同陣営内では前列、絶対的な左列の順とする。同じユニットの複数PSは定義順とする。
12. PSから新たなPSが誘発された場合は、既存の未処理候補より先に直ちに処理する。同じ戦闘ユニットが持つ同じPSは1行動につき1回だけ発動する。
13. PS連鎖が終わったら中断していたスキル処理へ戻る。解決途中で使用者が戦闘不能になった場合は、そのスキルの未解決の効果を中断する。
14. システムは、現在の行動より前に設定されていた行動単位クールタイムだけを減らし、行動単位のイベントと各時点の状態を記録する。現在の行動で設定したクールタイムは減らさない。
15. 戦闘不能になったユニットを行動順キューから即時除去する。
16. システムは1行動の終了後に勝敗を判定する。確定していなければ行動を継続する。
17. 行動速度が変化した場合、行動順キューに残る未行動ユニットだけを新しい速度順に並べ直す。予約済みの行動種別は変更しない。
18. 行動順キューが空になり、APまたは満タンのEXゲージが残る場合は、新しい行動順キューを生成して行動種別を予約する。
19. 使用可能な行動がなくなった場合は、現在のターンより前に設定されていたターン単位クールタイムだけを減らし、そのターンを終了する。現在のターンで設定したクールタイムは減らさない。
20. 敵陣営が全滅した場合、味方陣営の勝利とする。
21. 両陣営が同じ効果処理で全滅した場合も、味方陣営の勝利とする。
22. 味方陣営だけが全滅した場合、敗北とする。
23. 最終ターンの処理中に敵陣営が全滅した場合、勝利とする。
24. 最終ターンを終えて敵陣営が生存している場合、ターン上限による敗北とする。
25. システムは戦闘結果、最終状態およびイベントログを返す。

### 代替フロー

#### ASを使用できない

1. 最上位のASがクールタイム中、AP不足、対象不在、または発動条件を満たさない場合、次のASを判定する。
2. 使用可能なASが見つかるまで定義順に判定する。
3. 使用可能なASが1つもない場合、待機する。

#### PSが連鎖する

1. Battle Engineでドメインイベントが発生したとき、そのイベントを発動タイミングとして持つPSの条件を評価する。
2. 同じタイミングで条件を満たしたPSを1つの候補グループとし、先制攻撃を持つ候補を先に分けたうえで、各候補群を所有者の行動速度順に並べる。同速の場合は `Q-BTL-02` と同じ規則を適用する。
3. 先頭のPSが現在の1行動で未発動なら、PPなどの使用条件を確認して発動する。
4. PSによって別のPSが条件を満たした場合、新しく生じた候補グループを既存の未処理候補より先に直ちに処理する。
5. 新しい候補グループ内でも速度順と定義順を適用し、さらにPSが誘発された場合は再帰的に先に処理する。
6. 新しい候補グループをすべて解決した後、元の候補グループの処理へ戻る。
7. 現在の1行動で発動済みのPSは候補から除外する。
8. すべての候補がなくなるまで繰り返す。

同じユニットが持つ複数のPSが同時に候補となった場合は、スキルの定義順に処理する。

#### ターゲットを選択する

1. 俯瞰時の絶対左から `x=0,1,2`、敵後列、敵前列、味方前列、味方後列の順に `y=0,1,2,3` として、各配置位置を共通座標へ変換する。
2. 特別な対象指定がなければ、使用者から各候補までのマンハッタン距離 `|x1 - x2| + |y1 - y2|` を計算する。
3. 距離が小さい対象から並べる。
4. 同距離の場合は対象側の前列を優先し、さらに同順位なら俯瞰時の絶対的な左列を優先する。
5. スキルが要求する対象数まで順番に選択する。
6. 要求数より候補が少ない場合は、存在する対象だけにスキルを発動する。

味方陣営の前方はy座標が減る方向、敵陣営の前方はy座標が増える方向とする。「隣接」「目の前」は同じ陣営の盤面内だけを対象とし、陣営境界を越えない。ステルス適用後に代替対象が存在しない場合は `Q-TGT-05` の決定後に追記する。

#### スキル解決中に使用者が戦闘不能になる

1. 戦闘不能になるまでに解決済みの効果を記録する。
2. そのスキルに残っている未解決の効果を中断する。

#### 攻撃ダメージを計算する

1. 通常は攻撃を命中として扱う。暗闇や特別な回避効果がある場合だけ、その効果に従ってMISSを判定する。
2. 攻撃力から防御力を引き、0未満なら基礎ダメージを0とする。
3. スキル威力、属性相性、会心、被ダメージ補正などを適用する。途中結果は丸めない。
4. 会心判定では、会心率が0%未満なら0%、100%を超えるなら100%として確率を判定する。
5. 全補正を適用した最終結果の小数部分を切り捨てる。
6. 結果が1未満、または対象がダメージ無効効果を持つ場合も、ダメージを1とする。

#### シールドでダメージを受ける

1. 物理ダメージなら物理シールド、ENダメージならENシールドから先に差し引く。
2. 対応するタイプありシールドを使い切った後、タイプなしシールドから差し引く。
3. 通常のシールドを使い切った後、適用可能なサブユニットの耐久値から差し引く。
4. すべての適用可能なシールドを超えたダメージをHPから差し引く。

同じタイプのシールド値は付与時に既存の値へ加算する。

#### リンクダメージを適用する

1. リンク元で発生したダメージ量とダメージタイプを取得する。
2. リンクされた全対象へ、分割せず同量のリンクダメージをそれぞれ発生させる。
3. 各対象について、ダメージタイプに対応するシールド処理を行う。
4. リンクダメージから新たなリンクダメージは発生させない。

#### 継続ダメージを付与する

1. 効果付与時点の付与者の攻撃力を効果の状態として記録する。
2. 継続ダメージ発生時には、記録した攻撃力を使用する。
3. 付与後の攻撃力変化や付与者の戦闘不能は、記録済みの値へ影響しない。

#### 編成ボーナスを計算する

1. 5体の属性構成からスリーカード、フルハウス、フォーカード、ファイブカードを判定する。
2. コミカルをクレバー以外の各属性として評価し、最も高い編成ボーナスになる組み合わせを採用する。
3. クレバーの人数を数え、人数条件を満たしたすべてのクレバーボーナスを累積する。

#### 戦闘中ステータスを計算する

1. 同じ種類の重複ありバフ・デバフをすべて有効とする。
2. 同じ種類の重複なしバフ・デバフについて、効果量が最も大きい1件だけを有効とする。
3. メモリーの割合補正を重複ありバフとして加える。
4. すべての有効な割合補正を加算し、戦闘中割合補正を求める。
5. `基本値 × (1 + 編成補正 - 適性補正) × (1 + 戦闘中割合補正)` を計算する。
6. 乗算結果へメモリーの固定値補正を加える。

#### 行動単位効果期間を更新する

1. 行動終了時に、行動者自身を対象とする行動単位効果を取得する。
2. 現在の行動で付与された効果を除外する。
3. 残った各効果インスタンスの残り回数を1減らす。
4. 0になった効果を即時に失効させる。
5. 重複なしの最強効果が失効した場合、残っている同種効果から次に強い効果を有効にする。
6. 影響するステータスを再計算し、失効イベントと状態差分を記録する。

重複あり・重複なしのどちらも効果期間は個別に保持する。

#### ターン単位効果期間を更新する

1. ターン終了時にターン単位効果を取得する。
2. 現在のターンで付与された効果を除外する。
3. それ以前に付与された各効果インスタンスの残り回数を1減らす。
4. 0になった効果を即時に失効させる。
5. 重複なしの最強効果が失効した場合、残っている同種効果から次に強い効果を有効にする。
6. 影響するステータスを再計算し、失効イベントと状態差分を記録する。

#### チャージする

1. スキルを使用してチャージを開始し、誘発したPSまで解決して1行動を終了する。
2. 原則として次の行動機会にチャージ効果を発動する。
3. チャージ効果と、それにより誘発したPSまで解決して別の1行動を終了する。

#### 行動速度が変化する

1. 現在の1行動を最後まで解決する。
2. 現在の行動順キューに残っている未行動ユニットだけを新しい速度順に並べ直す。
3. 各ユニットに予約済みのASまたはEXは変更せずに保持する。
4. 更新後の行動順で処理を継続する。

#### キュー決定後にEXゲージが満タンになる

1. PSの発動などでEXゲージを最大値まで増加させる。最大値を超えた分は切り捨てる。
2. 現在の行動順キューでASが予約されている場合は、予約を変更せずASを使用する。
3. 現在のキューが空になり、次の行動順キューを生成する時点でEXを予約する。途中で速度変化による行動順再計算が発生しても、現在のAS予約は変更しない。

#### EXスキルを使用する

1. APを消費せず、EXゲージを全量消費してEXスキルを使用する。
2. EX使用後もAPが残っている場合、次に生成される行動順キューでASを予約できる。

#### 行動できない

気絶、凍結などによりスキルを使用できない場合、待機とリソース消費を記録する。APが1以上ならAPを1消費する。APが0かつEXゲージ満タンで行動順へ入った場合は、EXゲージを全量消費して待機する。

### 事後条件

- 戦闘結果が勝利または敗北のいずれかで確定している。
- 戦闘の最終状態が返却可能である。
- 行動単位の出来事を時系列に追跡できる。
- ログ中の各状態から、そのイベントによる変化を確認できる。

### 出力

最低限、次の情報を返す。

- 勝敗
- 終了理由：敵全滅、味方全滅、同時全滅、ターン上限
- 終了ターン
- 両陣営の最終状態
- 発生順に並んだイベントログ
- イベントごとの状態または状態差分

状態全体を毎回出力するか、差分と定期スナップショットを組み合わせるかはAPI設計で決定する。

## UC-02 戦闘条件の不備を通知する

### 目的

実行できないリクエストを戦闘開始前に検出し、問題箇所をクライアントへ通知する。

### エラー候補

- 存在しないユニットID
- 5体を超える編成
- 盤面外の配置
- 同一陣営内の配置重複
- 1未満または99を超える規定ターン数
- 存在しないメモリーID
- いずれかの編成でメモリーの指定数が6件を超えている

### 事後条件

- 戦闘は開始されない。
- 検出できた不備を、対象フィールドと理由が分かる形式で返す。

## 対象外ユースケース

- ユニット、スキル、メモリーの登録・更新・削除
- 戦闘途中でのプレイヤー入力
- 戦闘の一時停止と再開
- 過去の戦闘の永続化と検索
- 同一乱数系列を用いた完全な戦闘再現
