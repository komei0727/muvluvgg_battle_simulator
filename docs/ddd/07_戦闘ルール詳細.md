# 戦闘ルール詳細

## 目的

本書は、[06\_戦闘状態遷移.md](./06_戦闘状態遷移.md) の各遷移から呼び出す判定・計算ルールを、実装とテストが可能な粒度で定義する。

各ルールには `R-領域-連番` 形式のルールIDを付ける。テストケース、ドメインイベントおよび後続の実装設計から、このIDを参照できるものとする。

## ルールの優先順位

競合する記述がある場合は、次の順に優先する。

1. [02\_仕様確認事項.md](./02_仕様確認事項.md) の決定事項
2. 本書の詳細ルール
3. スキル・効果定義に委ねられた項目の個別指定
4. [戦闘システム.md](./戦闘システム.md) の基本仕様
5. デフォルトルール

スキル・効果定義による個別指定は、ターゲット方式や効果量など、その定義に委ねられた項目だけを上書きする。最低1ダメージ、PSの1解決スコープ1回制限などの全体不変条件は、明示的に上書きを許可した仕様がない限り変更しない。

## 共通の数値規則

### R-NUM-01 割合の内部表現

- 設計書の100%を内部値 `1.0` として扱う。
- 25%加算は `+0.25`、5%減算は `-0.05` とする。
- ダメージの最終結果を求めるまで、途中値を丸めない。
- ダメージ以外のステータス計算値も途中で丸めず保持する。

### R-NUM-02 整数化

- 最終ダメージは全補正適用後に小数部分を切り捨てる。
- HP、AP、PP、EXゲージへ適用する `FormulaDefinition` の結果は、適用直前に小数部分を切り捨てる。
- 割合補正、DamageModifier、HealingModifier の倍率値は途中で丸めない。
- AP、PP、EXゲージを負数にしない。
- 回復量、シールド付与量、リソース増減量は適用直前に小数部分を切り捨てる。

### R-NUM-03 確率判定

- 確率判定は `RandomSource` を通して行う。
- 会心率のステータス値自体には上限・下限を設けない。
- 会心判定時の実効会心率だけを0～100%へ補正する。
- 0%は必ず失敗、100%は必ず成功とする。

### R-NUM-04 Formula評価

- `FormulaDefinition` は数値を返し、状態変更を行わない。
- `SUM`、`MIN`、`MAX`、`CLAMP` などの合成Formulaは、子Formulaの評価結果を丸めずに合成する。
- `DAMAGE_DEALT_RATIO` と `DAMAGE_RECEIVED_RATIO` は、直前の確定済みダメージ結果を参照する。
- `MARKER_COUNT_SCALE` は評価時点の `MarkerState.stackCount` を参照する。
- Formula が参照する対象、binding、イベントpayload、直前結果が存在しない場合、その定義は戦闘開始前のCatalog検証またはpreflightで拒否する。

## 編成の検証

### R-FRM-01 編成人数

- 味方・敵それぞれ1～5体とする。
- 0体または6体以上の編成は戦闘開始前に拒否する。

### R-FRM-02 配置

- 各陣営の盤面は横3列、縦2列とする。
- 列は絶対左から左、中央、右とする。
- 行は相手陣営に近い側を前列、遠い側を後列とする。
- 同じ陣営内で同じ配置位置を重複させない。

### R-FRM-03 ユニット重複

- 同じ `UnitDefinitionId` を同一陣営へ複数指定できる。
- 各参加枠へ異なる `BattleUnitId` を割り当てる。
- スキル状態、HP、効果、イベントログは `BattleUnitId` 単位で管理する。

### R-FRM-04 メモリー指定

- 各編成に0～6件のメモリーIDを指定できる。
- 存在しないメモリーIDまたは7件以上の指定を拒否する。
- メモリーの発動条件を満たさなくても入力エラーにはせず、効果を適用しない。

### R-FRM-05 規定ターン数

- 1～99の整数とする。
- 範囲外の値を戦闘開始前に拒否する。

### R-FRM-06 Capability preflight

1. 編成で参照する Unit、Skill、EffectAction、Memory を `BattleDefinitionSet` から推移的に列挙する。
2. 各定義の `requiredCapabilities` を収集する。
3. 収集した Capability ID が `capabilities.json` に存在しない場合は戦闘開始前に拒否する。
4. Capability の `runtimeStatus` が `IMPLEMENTED` 以外の場合は戦闘開始前に拒否する。
5. 拒否時は対象 Capability ID と、それを要求した定義IDをエラーへ含める。

未実装 Capability を持つ定義を実行時に黙って無視したり、近似ルールで処理してはならない。

## 共通座標と配置

### R-POS-01 共通座標への変換

列座標：

| 列       |   x |
| -------- | --: |
| 絶対左列 |   0 |
| 中央列   |   1 |
| 右列     |   2 |

行座標：

| 陣営・行 |   y |
| -------- | --: |
| 敵後列   |   0 |
| 敵前列   |   1 |
| 味方前列 |   2 |
| 味方後列 |   3 |

### R-POS-02 前方

- 味方の前方はy座標が減る方向とする。
- 敵の前方はy座標が増える方向とする。
- 絶対左列は陣営の向きにかかわらずx座標が小さい側とする。

### R-POS-03 距離

使用者 `(x1, y1)` と対象 `(x2, y2)` のマンハッタン距離を次で求める。

```text
distance = |x1 - x2| + |y1 - y2|
```

## 編成ボーナス

### R-BON-01 通常属性の役判定

編成が5体の場合、クレバーを除く属性構成について次の最も高い役を一つ適用する。

| 優先順位 | 役             | 条件                 | 攻撃力 |   HP |
| -------: | -------------- | -------------------- | -----: | ---: |
|        1 | ファイブカード | 同属性5体            |   +25% | +25% |
|        2 | フォーカード   | 同属性4体            |   +15% | +20% |
|        3 | フルハウス     | 同属性3体と別属性2体 |   +15% | +15% |
|        4 | スリーカード   | 同属性3体            |   +10% | +10% |

条件を満たさない場合、通常属性の編成ボーナスは0とする。4体以下の編成では役判定を行わない。

### R-BON-02 コミカル

1. 各コミカルをクレバー以外の属性として割り当てた全候補を評価する。
2. `ファイブカード > フォーカード > フルハウス > スリーカード > 役なし` の順で最も高い候補を採用する。
3. 採用した役の補正だけを適用する。
4. コミカルをクレバーとして扱わない。

### R-BON-03 クレバー

クレバーの人数が条件を満たしたすべての段階を累積する。

| 人数条件 | 追加補正           |
| -------- | ------------------ |
| 1人以上  | 防御力+30%         |
| 2人以上  | 攻撃力+10%、HP+10% |
| 3人以上  | 会心率+15%         |
| 4人以上  | 攻撃力+15%、HP+15% |
| 5人以上  | 攻撃力+25%、HP+25% |

例としてクレバー3人の場合、防御力+30%、攻撃力+10%、HP+10%、会心率+15%をすべて適用する。

## 戦闘中ステータス

### R-STA-01 基本式

対象ステータスごとに次を計算する。

```text
戦闘中ステータス =
  基本値
  × (1 + 編成補正 - 適性補正)
  × (1 + 戦闘中割合補正)
  + メモリー固定値補正
```

- 適正外配置によるHP、攻撃力、防御力の適性補正は `0.05` とする。
- 適正外配置の影響を受けないステータスでは適性補正を0とする。
- 編成補正、戦闘中割合補正、メモリー固定値補正は対象ステータスごとに集計する。

### R-STA-02 重複あり効果

- 同種の重複ありバフ・デバフをすべて有効とする。
- 有効な効果量を符号付きで加算する。
- メモリーの割合補正を重複ありバフとして加算する。

### R-STA-03 重複なし効果

- `EffectKindKey` が同じ重複なし効果を一つのグループにする。
- グループ内で効果の強度が最も大きい1件だけを計算へ採用する。バフは正の効果量、デバフは弱化量の大きさで比較する。
- 同じ絶対値の候補が複数ある場合も計算結果は同じため、効果インスタンスは変更せず最初に付与されたものを代表として扱う。
- 採用されない効果も、期間経過や解除に備えて保持する。

### R-STA-04 再計算

次の状態変更後、影響を受ける戦闘中ステータスを再計算する。

- バフ・デバフの付与、更新、解除、失効
- メモリー効果の有効・無効条件の変化
- その他、ステータス補正条件の変化

行動速度が変化した場合は、現在の1行動とPS/Memory連鎖完了後に未行動者を並べ直す。

## 行動順

### R-ORD-01 行動可能判定

戦闘可能であり、次のいずれかを満たすユニットを新しい行動順キューの候補とする。

- APが1以上
- EXゲージが満タン
- 凍結などで阻害されていないチャージ効果が発動待ち

### R-ORD-02 並べ替え

次の比較キーを上から順に適用する。

1. 現在の行動速度：降順
2. 陣営：味方、敵
3. 行：前列、後列
4. 列：絶対左、中央、右

この比較によって行動順を一意にする。

### R-ORD-03 行動種別予約

- キュー生成時点でEXゲージが満タンならEXを予約する。
- 満タンでなければASを予約する。
- キュー生成後にEXゲージが満タンになっても予約を変更しない。
- 速度変化による並べ替えでも予約を変更しない。
- チャージ効果発動は実効行動決定時にAS／EX予約より優先する。

### R-ORD-04 速度変化

1. 現在の1行動とPS/Memory連鎖を完了する。
2. キューから処理済み・戦闘不能のユニットを除く。
3. 残った未行動者を `R-ORD-02` で並べ直す。
4. 各予約のAS／EX種別を維持する。

## AS・EX・待機の選択

### R-ACT-01 実効行動の優先順

1. 気絶中：待機。チャージ中ならチャージをキャンセルする。
2. 凍結中：待機。チャージを維持する。
3. 発動可能なチャージ効果あり：チャージ効果を発動する。
4. EX予約：EXスキルを使用する。
5. AS予約：`R-ACT-02` でASを選ぶ。

### R-ACT-02 AS選択

ASを定義順に評価し、最初に次の条件をすべて満たしたものを選ぶ。

- クールタイムが0
- 必要APを保有
- 発動条件を満たす
- 必要な対象候補が1体以上存在する
- 気絶、凍結などによって使用を禁止されていない

候補がなければ待機する。

### R-ACT-03 リソース消費

| 行動                     | 消費           | EXゲージ増加 |
| ------------------------ | -------------- | ------------ |
| AS                       | スキル定義のAP | 消費APと同量 |
| PS                       | スキル定義のPP | 消費PPと同量 |
| EX                       | EXゲージ全量   | なし         |
| 通常待機                 | AP 1           | 1            |
| AP 0・EX満タンで行動不能 | EXゲージ全量   | なし         |
| チャージ効果発動         | なし           | なし         |

AS・PS・EXのスキル定義コスト（AP／PP／EXゲージ）は1以上とする。コスト0のAS・PS・EXは存在しない（Catalog検証で拒否する）。

EXゲージ増加後はユニット固有の最大値で打ち止めとし、超過分を保持しない。

### R-ACT-04 リソース変更イベント

- AP、PP、EXゲージが変化した場合、変化後に `ResourceChanged` を発行する。
- 変化量が0の場合は `ResourceChanged` を発行しない。
- 消費と増加が同じ行動で発生する場合、消費を先に適用し、その後に増加を適用する。
- `MODIFY_RESOURCE` によるリソース変更も同じイベントを使用する。
- M7でリソース獲得量Modifier（`RESOURCE_GAIN_MOD`）が対象ユニットに有効な場合、基礎増加量（`ResourceChanged.baseDelta`）を算出した後、有効なModifierをすべて合成してから適用し、最後にcapacityで打ち止めてから `ResourceChanged`（`delta`に最終量を記録）を発行する。合成順、丸め、上限適用順の詳細はM7-002（Issue #185）で確定する。Modifierが存在しない場合はこの節の規則のみを適用するが、capacity打ち止めが発生する場合は`baseDelta`と`delta`が一致しない。
- capacity打ち止めによって`delta`が0になる場合（例：EXゲージが既に上限で正の`baseDelta`が全量破棄される場合）も「変化量が0の場合は`ResourceChanged`を発行しない」規則をそのまま適用するが、`ExtraGaugeOverflowDiscarded`は`delta`が0でも常に発行し、`baseDelta`を含む要求増加量・実増加量・破棄量を保持する。

## ターゲット候補

### R-TGT-01 候補生成

1. スキル定義の対象陣営、対象種別、発動条件で候補を絞る。
2. 戦闘不能者を対象にできる明示的な指定がない限り、戦闘不能者を除く。
3. 自分自身だけを対象とするスキルでは使用者だけを候補にする。
4. 候補が0体なら、そのASを発動不能とする。

### R-TGT-02 デフォルト順

次の比較キーを上から順に適用する。

1. 使用者からのマンハッタン距離：昇順
2. 対象側の行：前列、後列
3. 対象の列：絶対左、中央、右

### R-TGT-03 最も遠い

`R-TGT-02` の並び全体を逆順にする。

### R-TGT-04 隣接

- 基準対象と同じ陣営の盤面内で、上下左右1マスを候補とする。
- 陣営境界を越えない。
- 斜め位置を含めない。

### R-TGT-05 目の前

- 基準対象と同じ陣営の盤面内で、同じ列の1マス前を候補とする。
- 基準対象が前列の場合は候補なしとし、スキルを発動不能とする。
- 陣営境界を越えない。

### R-TGT-06 列優先

1. 指定列にいる候補を先に並べる。
2. 左右列指定では、指定列からの列距離が小さい順に別列を並べる。
3. 前後列指定では指定列、反対列の順に並べる。
4. 同じ列優先度の候補は `R-TGT-02` で並べる。
5. 必要数に達するまで上から選ぶ。

右列優先の列順は右、中央、左となる。左列優先は左、中央、右となる。

### R-TGT-07 対象数不足

- 選択可能な候補が必要数より少ない場合、存在する候補だけを対象にする。
- 候補が1体以上ならスキル自体は発動する。
- 候補が0体ならAS選択時に発動不能として次のASを評価する。

### R-TGT-08 ステルス

1. 通常の対象順を決定する。
2. 第一優先対象がステルスを持つ場合に限り、ステルスを消費する。
3. その対象を候補順の最後へ移動する。
4. 移動後に代替対象が存在しない場合は、ステルスを消費したうえで元の対象へ発動する。
5. 第一優先対象でないステルス所持者は順序を変更しない。
6. 自分自身を対象とする自分のスキルには適用しない。
7. 条件によって対象が1体に限定されている場合は適用しない。

ステルス適用後に代替対象がいない場合も、スキルは不発にしない。

### R-TGT-09 TargetSelector 評価順

`TargetSelectorDefinition` は次の順で評価する。

1. `kind` に基づき初期候補を作る。
   - `SELF`: 使用者または効果発動者。
   - `TRIGGER_SOURCE`: trigger event の source。
   - `TRIGGER_TARGET`: trigger event の target。
   - `SELECT`: `side` で指定された陣営のユニット。
   - `BINDING_DERIVED`: `base` の対象から `area` または派生条件で候補を作る。
2. `includeDefeated: true` でない限り、戦闘不能者を除外する。
3. `filters` を定義順に適用する。`AND` / `OR` / `NOT` は入れ子条件として評価する。
4. `area` が指定されている場合、候補を範囲で絞る。
5. `order` を上から順に比較キーとして適用し、候補順を一意にする。
6. `count` が `ALL` でない場合、候補順の先頭から指定数を選ぶ。
7. 候補が0件かつ `fallback` がある場合、`fallback` を評価する。

`order` が省略された場合は `DEFAULT` とし、`R-TGT-02` と同じ距離、前列、絶対左列の順を使う。

### R-TGT-10 TargetBinding

- `EffectSequence` の開始時に、定義順で `targetBindings` を評価する。
- 評価結果は同じ `EffectSequence` 内の `TargetReference.kind: BINDING` から参照する。
- binding の評価後に戦闘状態が変化しても、同じ sequence 内の当該 binding は再評価しない。
- ただし、参照時点で対象が戦闘不能になっている場合、その `EffectAction` が戦闘不能者を対象にできる明示指定を持たない限り、その対象への適用をスキップする。
- `TargetBindingId` が存在しない参照は Catalog 検証で拒否する。

## スキル解決

### R-SKL-01 効果順

- `SkillDefinition.resolution.effectSequence` の `targetBindings` を定義順に評価する。
- `EffectSequence.steps` を定義順に解決する。
- 各 `EffectStep` と `EffectAction` の前後に必要なドメインイベントを発行する。
- イベントに反応したPS連鎖またはMemory triggeredEffectsを直ちに解決してから、次のstepへ進む。
- 使用者が戦闘不能になった場合、未解決効果を中断する。
- 解決済みの効果を巻き戻さない。

### R-SKL-02 複数対象

- `ACTION` step の対象が複数いる場合、bindingの対象順に処理する。
- 対象ごとの効果適用後に発生したPSまたはMemory triggeredEffectsを直ちに解決する。
- 使用者が途中で戦闘不能になった場合、残りの対象へ効果を適用しない。

### R-SKL-03 複数ヒット

- 各ヒットを独立したダメージイベントとして処理する。
- 暗闇によるスキル単位のMISS結果は全ヒットで共有する。
- MISSでなければ、対象固有の特別な回避、会心、ダメージ、シールド、戦闘不能、PS/Memory連鎖をヒットごとに解決する。
- 使用者が途中で戦闘不能になった場合、残りのヒットを中断する。

### R-SKL-04 クールタイム設定

- スキル使用開始時にクールタイムを設定する。
- 設定した行動・ターンの終了時には減らさない。
- 次回以降の自身の行動終了またはターン終了で1ずつ減らす。
- 残数が0になった後の次の行動機会から使用できる。

### R-SKL-05 チャージ

- チャージ開始時に元スキルのコストを消費し、クールタイムを設定する。
- チャージ開始を一つの行動として完了する。
- 次の発動可能な行動機会に、AS／EX予約より優先してチャージ効果を発動する。
- チャージ効果発動ではAP・EXゲージを消費しない。
- チャージ効果発動を別の一つの行動とする。
- 気絶付与時にチャージをキャンセルする。
- 凍結中はチャージを維持し、解除後の次の行動機会に発動する。

### R-SKL-06 ACTION step

1. step の `condition` を評価する。省略時は `TRUE` とする。
2. condition が false の場合、その step 全体をスキップする。
3. `target` で参照される対象集合を取得する。
4. 対象集合の順序に従って、各対象へ `actions` を定義順に適用する。
5. 1つの `EffectAction` 適用後に発生したイベントからPSまたはMemory triggeredEffectsが候補になった場合、直ちに解決してから次の action へ進む。

### R-SKL-07 BRANCH / RANDOM_BRANCH / REPEAT

- `BRANCH` は `condition` が true なら `thenSteps`、false なら `elseSteps` を定義順に解決する。
- `RANDOM_BRANCH` の `WEIGHTED_ONE` は weight に応じて1分岐だけを選び、選択結果をドメインイベントへ記録する。
- `RANDOM_BRANCH` の `INDEPENDENT` は branch 定義順に確率判定を行い、成功したbranchのstepsを定義順に解決する。
- `RANDOM_BRANCH` の乱数消費順はCatalog定義順とする。
- `REPEAT` は指定回数だけ `steps` を繰り返す。繰り返し途中で使用者が戦闘不能になった場合、残りの繰り返しを中断する。

### R-SKL-08 直前結果

- `LAST_RESULT` Condition と `DAMAGE_DEALT_RATIO` / `DAMAGE_RECEIVED_RATIO` Formula は、同じ解決スコープ内の直前に確定した `EffectAction` 結果を参照する。
- `BRANCH` や `REPEAT` の内側で発生した結果も、次の step から直前結果として参照できる。
- MISS、付与拒否、対象不在などで効果が適用されなかった場合も、結果種別を持つ直前結果として記録する。

### R-SKL-09 クールタイム操作（`COOLDOWN_MANIPULATION`）

Issue #129。`EffectAction`から他スキルのクールタイムを短縮・リセットする。

- `EffectSequence`の定義順で操作を実行し、後続の`EffectAction`から変更後のクールタイム状態が観測できるようにする。
- `RESET`は対象スキルの残数を0にする。`REDUCE`は`amount`だけ減らし、0未満にはしない。
- 対象スキルがREADY（未登録、または残数が既に0）の場合、残数は変化しないためno-opとし、`CooldownReduced`/`CooldownCompleted`を発行しない。
- `R-SKL-04`の設定scope（設定した行動・ターンでは自然減算しない）は自身の行動・ターン終了時の減算にのみ適用され、`COOLDOWN_MANIPULATION`による明示操作は対象スキルが今回の行動・ターンで設定されていても適用する。
- 実際に残数が変化した場合は`CooldownReduced`と`StateDelta`を記録し、0になった場合は続けて`CooldownCompleted`も記録する。
- ACTION単位・TURN単位のどちらの対象スキルに対しても、対象スキル自身が持つ`unit`を維持したまま残数だけを変更する。
- 対象スキルは操作元の`EffectAction`を保有するUnitと同じUnitが所有するスキルに限る（Catalog検証で拒否する）。
- `R-ACT-02`「クールタイムが0」の判定は、この操作によって変更された残数を参照する。`R-PS-04`の発動直前再確認も同じ判定を再利用できる境界とする（PS候補エンジン自体はM6スコープ）。

## PS発動

### R-PS-01 発動タイミング照合

1. Battle Engineがドメインイベントを発行する。
2. PS Skill の `TriggerDefinition` のイベント種別と条件を照合する。
3. 発生源、対象、陣営、スキル種別などを `ConditionDefinition` で評価する。所有者から見た対象のFormation位置関係（例：目の前）と、現在のroot／parentイベントが示すBattle／Turn phase（開始・終了処理中かどうか）も、この`ConditionDefinition`評価に含める。
4. 条件を満たしたPSを同じイベントの候補グループにする。

PSの発動タイミングは固定一覧に限定せず、新しいドメインイベントへ対応づけられるものとする。

位置関係条件（`TRIGGER_POSITION_RELATION`）は、対象が戦闘不能または不在になった場合に不成立として扱う。除外タイミング条件（`TRIGGER_EXCLUSION_TIMING`）は、root/ancestorイベントがBattle開始・Turn開始・Turn終了の解決スコープに属する場合だけ不成立とし、同じeventTypeが通常の行動解決中に発生した場合は候補化する。固定のeventType分岐でこれらを実装せず、`ConditionDefinition`から評価する。

### R-PS-02 候補順

候補グループ内を次の順で並べる。

1. 所有者の現在行動速度：降順
2. 味方陣営、敵陣営
3. 前列、後列
4. 絶対左、中央、右
5. 同じユニットではPS定義順

### R-PS-03 同時発動制限

- 同じイベントで候補になった、同時発動制限を持つPSだけを一つの制限グループにする。
- 先制攻撃候補を通常候補より優先し、その中で `R-PS-02` が最上位となるPS一つだけを候補に残す。
- 除外されたPSは発動済みとして記録しない。
- 別の発動タイミングで再度条件を満たした場合は、1解決スコープ1回制限に抵触しない限り候補になれる。

### R-PS-04 発動直前確認

候補を処理する直前に次を再確認する。

- 所有者が戦闘可能
- 所有者がチャージ中でない
- 必要PPを保有
- クールタイムが0
- 発動条件が現在も成立
- 現在の解決スコープで未発動

いずれかを満たさなくなった候補を破棄する。

### R-PS-05 発動と再入防止

1. `BattleUnitId + SkillDefinitionId` を発動済み集合へ追加する。
2. PPを消費し、消費量と同量だけEXゲージを増やす。
3. クールタイムを設定する。
4. `PassiveActivated` を発行する。
5. `EffectSequence` を `R-SKL-01` から `R-SKL-08` に従って解決する。
6. 完了したら `PassiveResolved` を発行する。

発動済み集合への追加をPP消費やイベント発行より先に行い、再入を防ぐ。

### R-PS-06 新規候補の即時処理

- PS解決中に新しい候補グループが生じた場合、既存の未処理候補より先に処理する。
- 新しいグループを候補スタックの先頭へ積む。
- 新しいグループ内でも `R-PS-02` を適用する。
- 新しいグループの解決後、元のグループの続きへ戻る。

### R-PS-07 1解決スコープ1回

- ユニットの1行動では、同じPSを1回だけ発動できる。
- ターン開始・終了など行動外では、そのトップレベルイベントの解決スコープで1回だけ発動できる。
- 解決スコープ終了時に発動済み集合を破棄する。

### R-PS-08 先制攻撃

- 同じイベントの候補グループでは、先制攻撃特性を持つPSを通常のPS候補より先に処理する。
- 先制攻撃候補が複数ある場合は `R-PS-02` の行動速度、陣営、配置、定義順を適用する。
- 先制攻撃は発動条件、発動直前確認、同時発動制限、1解決スコープ1回制限を無視しない。
- 先制攻撃PSから新しい候補が生じた場合も `R-PS-06` に従い、新しい候補グループを直ちに処理する。
- 先制攻撃候補をすべて処理した後、同じ候補グループに残る通常候補へ戻る。

## Memory発動

### R-MEM-01 triggeredEffects 候補化

1. Battle Engineがドメインイベントを発行する。
2. 編成に指定された Memory の `triggeredEffects` をAPI指定順に走査する。
3. `TriggerDefinition.eventType` と `ConditionDefinition` を評価する。
4. 条件を満たした triggeredEffect を同じイベントのMemory候補グループにする。

Memory triggeredEffects はPP、クールタイム、先制攻撃、1解決スコープ1回制限を持たない。

### R-MEM-02 候補順

Memory候補グループ内を次の順で並べる。

1. APIリクエストで指定された Memory の順序。
2. 同一 Memory 内の `triggeredEffects` 定義順。

同じイベントでPS候補とMemory候補が両方存在する場合、PS候補を先に解決し、その後Memory候補を解決する。PS解決中に新しいMemory候補が発生した場合は、その新しい候補を現在のPS候補グループより後、未処理の親Memory候補より前に積む。

### R-MEM-03 戦闘開始時Memory

- 戦闘開始時Memoryは `BattleStarted` イベントで候補化する。
- `BattleStarted` に対するMemory効果は、初回ターン開始前、行動順キュー生成前に解決する。
- `APPLY_STAT_MOD` を含む Memory の `triggeredEffects` は、通常スキル由来の `APPLY_STAT_MOD` と同じく `AppliedEffect` として保持し、`CombatStatCalculator` で再計算する。
- このルールはM7のMemory発動エンジン実装（`BattleStarted` イベント処理）の責務であり、`FormationFactory` による初期 `BattleParty` 生成では `triggeredEffects` を解決しない。`FormationFactory` はMemory補正を一切計算せず、開始ステータスは編成ボーナスと配置適性のみで確定する。

### R-MEM-04 Memory EffectSequence

Memory の `EffectSequence` はスキルと同じく `R-SKL-01` から `R-SKL-08` に従って解決する。ただし使用者はMemoryを指定した陣営を source side とし、対象参照の `SELF` は使用できない。具体的な発生源 BattleUnit が必要なEffectActionをMemoryから使用する場合は、Catalog検証またはpreflightで拒否する。

## EffectAction解決

### R-ACTN-01 共通

1. EffectAction の `requiredCapabilities` が実装済みであることをpreflightで確認する。
2. EffectAction の対象が戦闘不能であり、かつ戦闘不能者を対象にできる明示指定がない場合は適用しない。
3. EffectAction の payload に含まれる `FormulaDefinition` を評価する。
4. EffectAction の種類に応じて HP、リソース、`AppliedEffect`、`MarkerState`、`ShieldState` を変更する。
5. 状態変更後、対応するドメインイベントを発行する。
6. イベントに反応したPSまたはMemory triggeredEffectsを解決してから、次のEffectActionへ進む。

### R-ACTN-02 即時効果

- `DAMAGE` は `R-HIT-*`、`R-CRT-*`、`R-DMG-*`、`R-SHD-*` に従って解決する。
- `HEAL` は `R-HEAL-*` に従ってHPを回復する。
- `MODIFY_RESOURCE` は対象リソースへ整数化済みFormula結果を適用し、0以上かつ現在最大値以下に丸める。
- `REMOVE_EFFECTS` は対象カテゴリ、解除数、優先順に従って `AppliedEffect` または `MarkerState` を解除する。

### R-ACTN-03 継続状態の付与

- `APPLY_STAT_MOD`、`APPLY_DAMAGE_MOD`、`APPLY_HEALING_MOD`、`APPLY_STATUS`、`APPLY_SHIELD`、`EFFECT_IMMUNITY`、`APPLY_DEATH_SURVIVAL`、`APPLY_TARGET_REDIRECT`、`APPLY_COVER`、`APPLY_REFLECT`、`APPLY_DAMAGE_LINK` は `AppliedEffect` として保持する。
- `APPLY_MARKER` は `MarkerState` として保持する。Marker に関連する継続効果がある場合は linkedEffectGroup で関連づける。
- 新規付与が無効効果によって拒否された場合、状態を変更せず、付与拒否イベントを記録できる。
- 付与後、影響するステータス、シールド、行動速度、対象選択条件を再評価する。

## 命中判定

### R-HIT-01 通常命中

- 通常の命中率・回避率を設けない。
- 暗闇または特別な回避効果がなければ必ず命中する。

### R-HIT-02 特別な回避効果

- 対象が有効な回避効果を持つ場合、その効果定義に従って回避を判定する。
- 攻撃が必中なら回避効果自体を発動させない。
- 対象がチャージ中なら自身の回避効果を発動させない。

### R-HIT-03 暗闇

1. スキル使用ごとに1回、使用者に付与された暗闇を付与順に取得する。
2. 各暗闇の指定確率でMISS判定を行う。
3. いずれか一つの暗闇でMISSになった場合、そのスキル全体をMISSとして扱う。
4. 複数対象・複数ヒットでも同じ判定結果を使用する。
5. 暗闇同士の確率を加算・乗算して一つの確率にまとめない。
6. 必中を持つスキルにも適用する。

MISSの場合、対象へのダメージと効果を適用しない。ただしMISSを契機とするPSは、対応するイベント定義があれば発動候補になれる。

## 会心判定

### R-CRT-01 実効会心率

```text
実効会心率 = min(100%, max(0%, 戦闘中会心率))
```

### R-CRT-02 会心倍率

会心が発生した場合、次の倍率を使用する。

```text
会心倍率 = 150% + 会心ダメージボーナス
```

会心が発生しない場合の倍率は100%とする。

## 属性相性

### R-ATR-01 有利属性

| 攻撃側       | 有利な防御側 |
| ------------ | ------------ |
| アグレッシブ | シャイ       |
| シャイ       | キュート     |
| キュート     | スマート     |
| スマート     | アグレッシブ |
| コミカル     | クレバー     |
| クレバー     | コミカル     |

### R-ATR-02 属性倍率

- 有利属性の場合は `125% + 属性相性ボーナス` とする。
- 有利属性でない場合は100%とする。
- 属性相性ボーナスは有利属性の場合だけ加算する。

## ダメージ計算

### R-DMG-01 基本ダメージ

```text
実効防御力 = 防御側の戦闘中防御力 × (1 - defenseIgnoreRate)
基礎ダメージ = max(0, 攻撃側の戦闘中攻撃力 - 実効防御力)

計算ダメージ =
  基礎ダメージ
  × スキル威力
  × 属性倍率
  × 会心倍率
  × 与ダメージ倍率
  × 被ダメージ倍率
  × Action内追加ダメージ倍率
```

`FormulaDefinition.kind: SKILL_POWER` は、スキル威力として上式へ入る。`damageModifiers` はその `DAMAGE` EffectAction だけに適用する Action内追加ダメージ倍率として扱う。

### R-DMG-02 最終化

1. 全倍率適用後に小数部分を切り捨てる。
2. ダメージ無効効果がある場合も結果を1とする。
3. 計算結果が1未満の場合も1とする。
4. 最終ダメージを0未満にしない。

最低1ダメージは、ダメージを発生させる攻撃にだけ適用する。ダメージ効果を持たないスキルから1ダメージを発生させない。

### R-DMG-03 防御貫通

- `defenseIgnoreRate` は基本ダメージ式で参照する防御力を割合で無視する。
- `shieldIgnoreRate` は最終ダメージのうち、シールドを経由せずHPへ直接適用する割合を表す。
- `damageReductionIgnoreRate` は被ダメージ軽減効果のうち、軽減方向の補正だけを割合で無視する。
- いずれも `0` なら通常処理、`1` なら全量無視とする。
- `shieldIgnoreRate` によるHP直接適用量は `floor(最終ダメージ × shieldIgnoreRate)` とし、残りをシールド適用対象とする。

### R-DMG-04 DamageModifier

- 与ダメージ補正は攻撃側に有効な `APPLY_DAMAGE_MOD direction: OUTGOING` を集計する。
- 被ダメージ補正は防御側に有効な `APPLY_DAMAGE_MOD direction: INCOMING` を集計する。
- `damageType` が指定されている補正は、対象のダメージタイプと一致する場合だけ適用する。
- 補正値は符号付き割合として加算し、倍率は `1 + 合計補正` とする。
- 倍率が0未満になる場合は0として扱う。
- `damageReductionIgnoreRate` は、被ダメージ補正のうち負の補正だけに適用する。

### R-DMG-05 ダメージイベント順

`DAMAGE` EffectAction は対象ごと、ヒットごとに次の順で処理する。

1. `UnitBeingAttacked` を発行する。
2. 命中判定を行う。
3. MISSでなければ会心判定を行う。
4. `DamageWillBeApplied` を発行する。
5. 防御介入系状態を `R-INT-01` の順で評価する。
6. ダメージを計算し、`DamageCalculated` を発行する。
7. シールド、サブユニット、HPへ適用し、`DamageApplied` を発行する。
8. HPが0になった場合は `UnitDefeated` を発行する。
9. リンクダメージ、反射などの追加ダメージがある場合は、各Capabilityのルールに従って追加イベントを発行する。

## 回復計算

### R-HEAL-01 即時回復

1. `FormulaDefinition` を評価する。
2. 回復量を適用直前に小数部分切り捨てで整数化する。
3. 回復量が0未満の場合は0とする。
4. 対象のHPを最大HPを超えない範囲で増加させる。
5. `HealApplied` を発行する。

`overheal` は初期実装では `DISCARD` のみとし、最大HPを超えた分は破棄する。

### R-HEAL-02 HealingModifier

- 回復量補正は `APPLY_HEALING_MOD` を集計する。
- 補正値は符号付き割合として加算し、倍率は `1 + 合計補正` とする。
- 倍率が0未満になる場合は0として扱う。
- 回復量Formulaの結果に HealingModifier 倍率を適用し、その後で整数化する。

### R-HEAL-03 継続回復

- `APPLY_CONTINUOUS_HEAL` は `AppliedEffect` として保持する。
- 指定された eventType と対象条件を満たしたとき、`R-HEAL-01` と同じ手順で回復する。
- 継続回復のDuration消費は `R-EFF-*` に従う。

## シールドとHPへの適用

### R-SHD-01 シールドプール

- 物理シールド、ENシールド、タイプなしシールドを別々に保持する。
- 同じタイプのシールド付与値を加算する。
- 個別消滅条件を持つ付与元は `AppliedEffect` として保持し、有効な合計値を算出する。

### R-SHD-02 適用順

ダメージを次の順に適用する。

1. `shieldIgnoreRate` 分をHPへ直接適用する。
2. 残りをダメージタイプに対応するタイプありシールドへ適用する。
3. さらに残った分をタイプなしシールドへ適用する。
4. さらに残った分をサブユニットへ適用する。
5. さらに残った分をHPへ適用する。

対応しないタイプありシールドへダメージを適用しない。

### R-SHD-03 超過ダメージ

- シールド値を超えた残りを次の適用先へ渡す。
- HPを0未満にせず、超過分を破棄する。
- HPが0になった時点で戦闘不能処理を行う。

## サブユニット

### R-SUB-01 シールド効果

- 通常シールドをすべて適用した後にサブユニットがダメージを受ける。
- 毒、炎上など、通常シールドで受けられないダメージはサブユニットでも受けない。
- サブユニットの残HPをシールド表示値へ合算できるが、内部状態は通常シールドと分ける。
- シールド解除・シールド無視の対象とする。

### R-SUB-02 追加効果

- 所持者の攻撃対象ごとに追加ダメージを1ヒット加える。
- 複数対象への攻撃では、各対象へ1ヒットずつ加える。
- 追加デバフが定義されている場合も対象ごとに適用する。
- 追加ダメージでは通常の防御力減衰を行わない。

サブユニット追加ダメージは次の式で算出する。

```text
追加ダメージ = サブユニット所持者の攻撃力 + 付与者の攻撃力 × スキル倍率 - 対象の防御力
```

最終ダメージの小数切り捨てと最低1ダメージは通常のダメージ規則に従う。

## 防御介入とリンクダメージ

### R-INT-01 防御介入順

`DamageWillBeApplied` 後、ダメージ確定前に防御介入系状態を次の順で評価する。

1. `APPLY_TARGET_REDIRECT`: 攻撃対象の引き寄せ、挑発。
2. `APPLY_COVER`: 肩代わり。
3. `APPLY_DAMAGE_LINK`: 継続リンク状態。
4. `APPLY_REFLECT`: 反射。
5. `APPLY_DEATH_SURVIVAL`: 致死耐え。

各介入は `requiredCapabilities` が実装済みの場合だけ実行できる。未実装の場合はpreflightで拒否する。

### R-INT-02 肩代わり

- 肩代わりが成立した場合、以後のダメージ計算とシールド適用の防御側を肩代わり者へ変更する。
- 元の攻撃対象を参照する後続効果は、EffectAction定義で明示されていない限り、最終的にダメージを受けた対象を参照する。
- 複数の肩代わり候補がある場合の優先順はCapability実装時に具体化する。

### R-INT-03 反射

- 反射は元ダメージの確定後、元ダメージの適用結果を巻き戻さずに発生する。
- 反射ダメージは新しいダメージとして扱うが、反射からさらに反射を発生させない。
- 反射ダメージは `isReflectedDamage=true` を持つ。

### R-LNK-01 リンク元

- 対象へ算出された最終ダメージをリンク元の量とする。
- シールド・HPへの振り分け前の量を使用する。
- 元ダメージの物理／ENタイプと、シールド適用可否を引き継ぐ。

### R-LNK-02 適用

- リンクされた全対象へ、リンク元と同量のダメージをそれぞれ発生させる。
- 対象数で分割しない。
- リンク先では属性、会心、ダメージ増減効果を再計算しない。
- リンク先ごとに対応するシールドとHPへ適用する。
- 元ダメージが毒・炎上などシールド対象外なら、リンク先でもシールド対象外とする。

### R-LNK-03 再リンク禁止

- リンクダメージに `isLinkedDamage=true` を付ける。
- `isLinkedDamage=true` のダメージから新たなリンクを発生させない。

## 継続ダメージ

### R-DOT-01 共通

- 付与時に付与者の攻撃力をスナップショットとして記録する。
- 付与後の攻撃力変化や付与者の戦闘不能は計算へ影響しない。
- 付与対象の行動開始時に発生する。
- ダメージ軽減・増加、属性相性の影響を受けない。
- 効果定義の物理／ENダメージタイプを使用する。
- 各継続ダメージの最終結果で小数部分を切り捨て、ダメージが1未満なら最低1とする。

### R-DOT-02 固定継続ダメージ

- スナップショット攻撃力と効果定義の倍率から固定ダメージを算出する。
- 対応するタイプありシールド、タイプなしシールド、HPの順で適用する。

### R-DOT-03 炎上

- 最大3つまで保持する。
- 各炎上は付与時攻撃力、効果量、期間を保持する。
- 対象が炎上を3つ保持している場合、各炎上インスタンスが発生させるダメージをそれぞれ2倍にする。
- 2倍処理は各インスタンスの最終結果を算出する前に適用し、合計値を後から2倍にしない。

### R-DOT-04 毒

```text
割合ダメージ = 現在HP × 毒効果率
上限ダメージ = 付与時攻撃力 × 100%
毒ダメージ = min(割合ダメージ, 上限ダメージ)
```

- 毒ダメージはシールドとサブユニットで受けない。
- 既存の毒へ再付与した場合、効果期間は長い方、効果量は大きい方を引き継いだ一つの毒を残す。
- 期間と効果量は別々の付与元から採用できる。

## 状態異常

### R-STS-01 共通

- 状態異常はデバフの一種とする。
- デバフ解除・デバフ無効の対象とする。
- 状態異常解除・状態異常無効は、状態異常として定義された効果だけを対象とする。

### R-STS-02 気絶

- AS、PS、EXスキルを新たに使用できない。
- 指定回数のアクティブ行動機会を待機すると解除する。
- APがあれば待機でAPを1消費する。
- AP 0・EX満タンならEXゲージを全量消費して待機する。
- 再付与時は残り回数が長い方を一つだけ残す。
- 付与時にチャージをキャンセルする。

### R-STS-03 凍結

- AS、PS、EXスキルを使用できない。
- 行動機会では待機する。
- 指定期間の経過、または新たな攻撃スキルによるダメージで解除する。
- 継続ダメージやデバフだけのスキルでは解除しない。
- 再付与時に期間延長や増幅率加算を行わない。
- チャージをキャンセルしない。
- 解除後の次の行動機会にチャージ効果を発動する。

攻撃スキルによるダメージで凍結を解除する場合、解除契機となったダメージを凍結効果定義の増幅率だけ増加させる。スキルに具体の増幅率が記載されていない場合は `+50%` を既定値とする。

### R-STS-04 暗闇

- AS、PS、EXスキルに適用する。
- 必中を無視してMISS判定を行う。
- 複数の暗闇を付与順に独立して処理する。
- 暗闇同士の確率を合算しない。

## 効果期間と解除

### R-EFF-01 期間単位

効果定義は次の期間単位と初期回数を持つ。

- 行動単位：付与対象自身の行動終了時に減らす。
- ターン単位：ターン終了時に減らす。
- 戦闘単位：戦闘終了まで保持する。
- ヒット単位：指定された与ヒットまたは被ヒットで減らす。
- スキル使用単位：指定されたスキル使用で減らす。

効果インスタンスごとに、残り回数、期間単位、owner、付与された行動ID、付与されたターン番号を保持する。`consumption`、`expiration`、`linkedEffectGroupId` は、回数による効果期間とは別に保持する。

### R-EFF-02 解除

1. 解除スキルの対象カテゴリを確認する。
2. バフ、デバフ、状態異常、シールドなど一致する効果を抽出する。
3. 解除数や優先順が定義されている場合はその指定に従う。
4. `EffectRemoved` を発行する。
5. ステータス、シールド、行動速度を再計算する。

### R-EFF-03 無効

- 無効効果が有効な間は、対象カテゴリの新規付与を拒否する。
- すでに付与済みの効果を解除するかどうかは、無効効果の定義に明示する。
- 付与拒否もドメインイベントとして記録できるものとする。

### R-EFF-04 行動単位期間の減算

1. 対象自身の行動終了時に、その対象へ付与された行動単位効果を取得する。
2. 効果の付与行動IDが現在の行動IDと同じ場合は減らさない。
3. 次回以降の対象自身の行動終了時に、各効果インスタンスの残り回数を1減らす。
4. 対象以外のユニットの行動では減らさない。
5. 残り回数が0になった時点で即時に失効させ、`EffectExpired` を発行する。
6. 失効後、影響するステータス、シールド、行動速度を再計算する。

### R-EFF-05 重複効果の期間

- 重複あり効果は、同種であっても効果インスタンスと残り期間を個別に保持する。
- 重複なし効果も、既存効果を上書きせず個別に保持する。
- 重複なし効果では、その時点で最も強い1件だけを計算へ採用する。
- 採用中の最強効果が失効・解除された場合、残っている同種効果を再評価する。
- 残存効果があれば次に強い1件を即時に有効化する。
- 次点効果の残り期間は、非採用中も通常どおり個別に減算する。

### R-EFF-06 ターン単位期間の減算

1. ターン終了時にターン単位効果を取得する。
2. 効果の付与ターン番号が現在のターン番号と同じ場合は減らさない。
3. 次のターン終了時から、各効果インスタンスの残り回数を1減らす。
4. 残り回数が0になった時点で即時に失効させ、`EffectExpired` を発行する。
5. 失効後、影響するステータス、シールド、行動速度を再計算する。
6. 重複なしの最強効果が失効した場合は、残存する次点効果を即時に有効化する。

### R-EFF-07 消費条件

- `NEXT_OUTGOING_ATTACK` は、効果ownerが次に攻撃を行い、命中判定まで到達した時点で消費する。
- `NEXT_INCOMING_ATTACK` は、効果ownerが次に攻撃対象になり、`UnitBeingAttacked` が発行された時点で消費する。
- `OUTGOING_HIT` は、効果ownerの攻撃ヒットがMISSでなく確定した時点で消費する。
- `INCOMING_HIT` は、効果ownerへの攻撃ヒットがMISSでなく確定した時点で消費する。
- `STATUS_BLOCKED` は、効果ownerへの状態付与が無効化された時点で消費する。
- 消費回数が0になった時点で即時に失効させ、`EffectExpired` を発行する。

同じイベントで時間制限と消費条件の両方が成立する場合、消費条件を先に評価し、その後に時間制限を評価する。どちらかで失効した効果は二重に失効処理しない。

### R-EFF-08 特殊失効条件

- `expiration.conditions` は関連するドメインイベント発行後、PS/Memory候補の抽出前に評価する。
- 条件が成立した効果は即時に失効させ、`EffectExpired` を発行する。
- 特殊失効後、影響するステータス、シールド、行動速度、TargetSelector結果を再評価する。

### R-EFF-09 linkedEffectGroup

- 同じ `linkedEffectGroupId` を持つ `AppliedEffect` と `MarkerState` は親子連動グループとして扱う。
- グループの親効果が失効・解除された場合、同じグループの子効果とMarkerも同時に失効させる。
- 子効果だけが消費条件で失効した場合、親効果は維持する。
- 同時失効では、子効果を先に失効させ、最後に親効果を失効させる。
- 失効イベントは各インスタンスごとに発行する。
- `AppliedEffect`同士だけで構成されるグループ（Marker不在）では、`DurationDefinition.linkedEffectGroupRole`（`PARENT`/`CHILD`）で親子を明示する。カスケード可否は失効理由（消費/時間制限/特殊失効）では判定しない — `PARENT`の失効は理由を問わず全体へカスケードし、`CHILD`単独の失効はカスケードしない。どちらもロールを持たないグループは対称にカスケードする（レビュー修正 PR #209、EFF-003）。

### R-EFF-10 Marker

- `APPLY_MARKER` の `ADD` は既存Markerのスタック数へ加算する。
- `KEEP_EXISTING` は既存Markerがある場合は変更せず、存在しない場合だけ新規付与する。
- `REFRESH` はスタック数を維持し、Durationを再設定する。
- `REPLACE` は既存Markerを新しい定義内容で置き換える。
- スタック数は0未満にせず、`stack.max` がある場合は上限を超えない。
- Marker が0スタックになった場合は解除し、`EffectRemoved` または `EffectExpired` を発行する。
- Marker の付与、更新、解除後は、Markerを参照するCondition、Formula、TargetSelectorの評価結果が変わり得る。

### R-EFF-11 RuntimeCounter

- RuntimeCounter は定義されたスコープ内で管理する。
- カウンター更新は、対象イベントの状態変更が確定した後、PS/Memory候補抽出の前に行う。
- `RUNTIME_COUNTER` Condition は更新済みの値を参照する。
- 更新は公開値（`value`）が変化した場合だけでなく、内部端数（`carry`、累計ダメージ閾値Counter専用）だけが変化した場合も`RuntimeCounterChanged`を発行する（レビュー再レビュー[P2]、Issue #143: `value`不変・`carry`不変の更新だけ何も発行しない）。可変状態の変化が必ずイベント列から追跡できるようにするため。
- 解決スコープ終了時にリセットするcounterは、PS/Memory候補スタックが空になった後、破棄してから`RuntimeCounterReset`を発行する。`RuntimeCounterReset`自身の候補解決によって同じスコープへcounterが再生成・更新された場合は、候補スタックが再び空になるたびに破棄・発行・候補解決を繰り返し、リセット対象counterが残らなくなってから解決スコープを終了する。この反復はPS発動済みGuard（`R-PS-07`）を経由しないため、実装は反復回数へ決定的な上限を設け、超過時はエラーとして検出する（`counterUpdates`が自身の`RuntimeCounterReset`を再生契機にする誤ったCatalog定義を想定、レビュー再レビュー[P1]）。
- Battle単位のcounterは戦闘終了まで保持する。

M6では`SkillRuntime`スコープ（PS発動条件用の最小範囲）だけを実装する。`Battle`／`BattleUnit`スコープは、それらを必要とするproduction Catalog定義が現れるまでFeature Complete必須対象に含めず、Catalogロード時点で明示的に拒否する（レビュー再レビュー[P2]、Issue #143：当初M6の対象としていたが、実装・検証コストに見合う対象行が現状ないためスコープを縮小した）。必要な定義を追加する際は、そのproduction経路と同じTaskで実装・検証する。`AppliedEffect`／`EffectSequence`スコープ（Marker・期間・消費条件に所有される汎用Counter管理）はM7のEFF-005（Issue #162）で扱う。いずれも同じ規則に従う。M6ではさらに次を扱う。

- `RUNTIME_COUNTER` Condition は `modulo` を持てる。`value MOD modulo == 0` を満たした更新でだけ成立する（N回ごと発動、`RUNTIME_COUNTER_MODULO`）。
- 累計ダメージ閾値Counterは、対象の最大HP比の閾値で加算する。1回の更新で閾値を複数回超えた場合はその回数だけ条件を成立させ、閾値未満の端数は次回の更新へ繰り越す（`CUMULATIVE_DAMAGE_THRESHOLD_TRIGGER`）。
- 複数回の条件成立は、候補抽出に複数回分の候補を生成することを意味しない。1回のイベントでCounterを更新した結果として成立する候補は、通常のPS候補抽出・発動済み制限・同時発動制限（`R-PS-02`〜`R-PS-08`）に従って1件ずつ処理する。

## ルールテストの基準

### 編成・ステータス

- コミカルの割り当て候補から最上位役を選べる。
- クレバー3人で1～3人条件の補正を累積する。
- 重複あり効果をすべて加算する。
- 重複なし効果では同種最大の1件だけを採用する。
- 適正外配置のHP、攻撃力、防御力だけを5%減らす。

### 行動・ターゲット

- 同速時に味方、敵、前列、絶対左列の順になる。
- 速度再計算でAS／EX予約を維持する。
- マンハッタン距離、前列、絶対左列で対象順が一意になる。
- 列優先で指定列が不足したとき隣の列から補う。
- 必要数より候補が少なくても存在する対象へ発動する。

### スキル・PS

- AP不足または対象不在のASを飛ばして次のASを選ぶ。
- TargetBindingをsequence開始時に評価し、同じsequence内で再評価しない。
- ACTION / BRANCH / RANDOM_BRANCH / REPEAT を定義順に解決する。
- 同じイベントでは先制攻撃PSを通常PSより先に処理し、先制攻撃同士を速度・陣営・配置・定義順で並べる。
- 新しく誘発されたPSを既存候補より先に解決する。
- 同じPSを1解決スコープで2回発動しない。
- 同時発動制限から最上位の1件だけを選ぶ。
- 使用者の戦闘不能後に残りのヒット・対象を処理しない。
- Memory triggeredEffectsをAPI指定順、同一Memory内定義順で解決する。

### ダメージ・効果

- 攻撃力が防御力以下でも最低1ダメージになる。
- ダメージ無効でも1ダメージになる。
- 最終結果だけを切り捨てる。
- 与ダメージ補正、被ダメージ補正、Action内追加ダメージ倍率を定義順に依存せず同じ結果で合成する。
- shieldIgnoreRate分をHPへ直接適用し、残りをシールドへ適用する。
- 回復量にHealingModifierを適用し、最大HPを超えた分を破棄する。
- タイプあり、タイプなし、サブユニット、HPの順で適用する。
- リンク先全員へ同量を適用し、再リンクしない。
- 毒の割合ダメージを付与時攻撃力で上限制限する。
- 行動単位効果が付与行動では減らず、対象自身の次回行動終了から減る。
- 重複あり効果の期間を個別に減らす。
- 重複なしの最強効果失効後に、残存する次点効果が有効になる。
- ターン単位効果が付与ターンでは減らず、次のターン終了時から減る。
- 消費条件、特殊失効、linkedEffectGroupの順序で効果が二重失効しない。
- Marker stackのADD、KEEP_EXISTING、REFRESH、REPLACEを区別できる。
- RuntimeCounter更新後の値でConditionを評価できる。
