# テスト戦略

## 目的

本書は、Battle Simulation Contextの仕様を継続的に検証するため、次を定義する。

- テストレベルと各レベルの責務
- ルールID、仕様確認ID、テストケースのトレーサビリティ
- 乱数、時刻、IDを制御した決定的なテスト方法
- ドメインルール、状態遷移、イベント、状態差分の検証方法
- API、Catalog、Worker Pool、負荷試験の範囲
- CIでの実行区分と品質ゲート

本書は [07\_戦闘ルール詳細.md](./07_戦闘ルール詳細.md) の109ルール、[08\_ドメインイベント.md](./08_ドメインイベント.md) 以降の設計、および [02\_仕様確認事項.md](./02_仕様確認事項.md) の決定・保留状況をテストへ対応づける。

## 品質目標

- 決定済みのドメインルールを変更時に即座に検知できる。
- 同じテスト入力と固定RandomSourceから同じ結果を得られる。
- 単独ルールの誤りと、複数ルールの接続順序の誤りを区別して診断できる。
- イベントログと状態差分から戦闘経過を一意に追跡できる。
- API、Worker、Catalogなど技術境界がドメインの意味を変えない。
- 未実装Capabilityを推測実装せず、該当定義を開始前に拒否できる。
- 性能劣化、無限連鎖、メモリー増大からサービスを保護できる。

## 基本方針

- 仕様の正本は設計書とルールIDであり、現在の実装結果をそのまま期待値にしない。
- ドメインルールは最も小さい責務のテストで詳細に検証する。
- Battle全体のテストでは、ルール間の順序、状態遷移、イベントを検証する。
- 外部ポートだけをテストダブルへ差し替え、Domain Entity同士を過剰にmockしない。
- 正常例だけでなく、境界値、不成立条件、順序逆転、再入防止を検証する。
- 大きなレスポンス全体のsnapshotだけに依存せず、意味のあるフィールドと不変条件をassertする。
- 発見した不具合には、修正前に最小の再現テストを追加する。
- 実行時間の長い負荷試験と、PRごとの高速な回帰テストを分離する。

## テストレベル

| レベル             | 主対象                                 | 外部依存                         | 主な目的                     | 実行頻度          |
| ------------------ | -------------------------------------- | -------------------------------- | ---------------------------- | ----------------- |
| Domain Unit        | 値オブジェクト、ポリシー、計算サービス | なし                             | 109ルールの個別検証          | 全PR              |
| Domain Scenario    | Battle集約、状態遷移、イベント連鎖     | 固定RandomSource                 | 複数ルールの接続順序         | 全PR              |
| Application Unit   | UseCase、事前検証、結果組み立て        | Portのfake／stub                 | ユースケース調整とエラー     | 全PR              |
| Catalog Contract   | JSON Schema、参照、Capability          | テストCatalog／実Catalog         | 定義データの構造・意味整合性 | 全PR              |
| API Contract       | DTO、JSON Schema、HTTP変換             | Fastify inject、fake Worker Pool | 外部契約の互換性             | 全PR              |
| Worker Integration | ESM Worker、Piscina、キャンセル        | 実Worker Thread                  | スレッド境界と障害処理       | main／PR選択      |
| End-to-End         | HTTPからBattle完了まで                 | 実Catalog、実Worker              | 全経路の成立確認             | main／リリース    |
| Property／Model    | 比較器、差分、リソースなど             | 生成データ                       | 広い入力空間の不変条件       | 全PR／nightly     |
| Load／Soak         | HTTP、Pool、Battle、メモリー           | 配備相当環境                     | 容量・性能・耐久性           | nightly／リリース |

Domain UnitとDomain Scenarioをテストの中心とする。End-to-Endだけでルールの網羅を目指さない。

## テストケース識別子

テストケースへ次の形式の安定したIDを付ける。

```text
<LEVEL>-<SUBJECT>-<NUMBER>
```

例：

- `UT-R-NUM-01-001`：ルール単体テスト
- `SCN-BTL-006`：Battle複合シナリオ
- `APP-SIM-004`：Applicationテスト
- `API-SIM-012`：API契約テスト
- `INT-WRK-003`：Worker結合テスト
- `PROP-ORD-002`：行動順のプロパティテスト

テスト名にはケースIDと期待する振る舞いを含める。

```text
UT-R-ACT-03-004: AP0・EX満タン・行動不能ならEXを全量消費して待機する
```

ケースIDはファイル名に必須とせず、テスト定義またはメタデータから検索可能にする。

## トレーサビリティ

### 追跡単位

次を相互に追跡する。

```text
仕様確認ID（Q-*）
  ↓
ルールID（R-*）
  ↓
テストケースID
  ↓
実装モジュール
```

通常のテストは `R-*` を直接参照する。複数ルールを確認するシナリオは、関連するすべてのルールIDをメタデータへ持つ。仕様決定の経緯確認が必要な場合だけ `Q-*` も関連づける。

### ルールカバレッジ台帳

機械判読可能な台帳をテストコード内に持つ。

```text
RuleTestCoverage {
  ruleId
  testCaseIds[]
  kinds: POSITIVE | NEGATIVE | BOUNDARY | PROPERTY | SCENARIO
}
```

CIは次を検証する。

- [07\_戦闘ルール詳細.md](./07_戦闘ルール詳細.md) から抽出した全109ルールIDが台帳に存在する。
- 台帳が未知または重複したルールIDを参照していない。
- 各ルールに少なくとも1件の実行対象テストがある。
- ルールの追加・削除時に台帳更新がない場合は失敗する。
- 未実装Capabilityは専用の拒否テストへ対応づいている。

Markdownのチェックリストだけで管理せず、テスト実行と台帳をCIで照合する。

### 必要なテスト種別

各ルールについて、適用可能な種別を選ぶ。

| ルール性質 | 必須テスト                                  |
| ---------- | ------------------------------------------- |
| 値域・上限 | 下限、上限、下限未満、上限超過              |
| 優先順     | 各優先キーが異なる例、完全同値の扱い        |
| 計算式     | 基本例、0、負方向補正、端数、最終丸め       |
| 発動条件   | 成立、不成立、成立後に再確認で不成立        |
| リソース   | 丁度消費、不足、0、最大値超過               |
| 期間       | 付与スコープ、次回、残1、残0、複数個別期間  |
| 連鎖       | 子なし、1段、複数段、循環候補、発動済み除外 |
| 状態遷移   | 正常遷移、不許可遷移、完了後更新            |
| エラー     | code、対象path／ruleId、Battle未開始        |

## 決定的なテスト

### SequenceRandomSource

テスト用 `RandomSource` は、指定した値を順に返す。

```text
SequenceRandomSource {
  values: number[]
  consumedValues: number[]
}
```

規則：

- 値はDomainのRandomSource契約範囲内とする。
- 値が不足した場合は即時にテストを失敗させる。
- 余った値も原則としてテスト失敗にし、想定外に乱数消費が減ったことを検出する。
- 呼び出し回数と順序をassertできるようにする。
- 値不足時にランタイム乱数へfallbackしない。

確率境界では「判定値が閾値の直前」「閾値と同値」「直後」をテーブル化する。会心率ステータスが0未満または100超でも、判定時の実効値だけが0～100へ補正されることを確認する。

### ID生成器

`BattleId`、`ActionId`、`SkillUseId`、`EffectInstanceId` などを予測可能に生成するテスト実装を使用する。

```text
battle-1
action-1
skill-use-1
effect-1
```

ID文字列そのものを検証目的にしないテストでは、IDの一意性と参照整合性だけをassertする。

### Clockと期限

実時間のsleepで期限テストを行わない。手動で進められるClockまたは期限判定fakeを使用する。

- 期限直前は継続する。
- 期限到達で `EXECUTION_TIMEOUT` になる。
- タイムアウトを `TURN_LIMIT_REACHED` へ変換しない。
- HTTPの強制キャンセルはWorker結合テストで別に扱う。

## テストデータ設計

### Builder

次のBuilderを用意する。

- `UnitDefinitionBuilder`
- `SkillDefinitionBuilder`
- `EffectDefinitionBuilder`
- `MemoryDefinitionBuilder`
- `FormationInputBuilder`
- `BattleScenarioBuilder`
- `CatalogBuilder`

Builderの既定値は「最小で妥当、特殊効果なし」とし、テスト対象以外の要因を減らす。ただし重要な既定値を隠さないよう、生成結果を読みやすいfixture名で表す。

例：

```text
unit({ hp: 1000, attack: 200, defense: 100, actionSpeed: 50 })
activeSkill({ apCost: 1, effects: [physicalDamage({ power: 100 })] })
```

### テストCatalog

用途別に分ける。

| Catalog            | 用途                                         |
| ------------------ | -------------------------------------------- |
| `minimal`          | 1対1の最小戦闘と単体シナリオ                 |
| `scenarios`        | PS、チャージ、状態異常など複合シナリオ       |
| `invalid`          | 参照切れ、重複ID、不正期間、未実装Capability |
| production Catalog | 実データ全体の構造・参照・Capability検証     |

ドメインルールテストをproduction Catalogの具体的なユニットへ依存させない。production定義の変更でルールテストの意図が変わることを防ぐ。

### 数値fixture

- 整数化前の端数が分かる値を使う。
- 0、1、最大値、最大値直前・直後を明示する。
- 浮動小数の途中値は必要に応じて許容誤差で比較する。
- 最終ダメージ、HP、AP、PP、EX、期間など整数値は完全一致で比較する。
- テスト期待値へproductionの計算関数を呼び出さない。

期待値の式をテストへ実装コードと同じ形で複製しすぎない。代表的な手計算例と、別形式の不変条件テストを組み合わせる。

## Domain Unitテスト

### 値オブジェクト

- 境界内の生成成功
- 境界外の生成失敗
- 等価性
- 不変性
- JSONやprimitiveへの変換境界

### 比較・選択ポリシー

行動順、PS順、ターゲット順は比較キーごとに独立したテーブルテストを作る。

```text
行動速度 desc
→ 陣営 ALLY before ENEMY
→ 前列 before 後列
→ column asc
```

同じ入力集合を並べ替えても結果順が同じになることをPropertyテストで確認する。入力配列順を暗黙のtie-breakerにしない。

### 計算サービス

- 各補正を一つずつ変えた例
- 複数補正を組み合わせた例
- 途中計算を丸めない例
- 最終値だけ切り捨てる例
- 最低値・上限値を適用する例
- 適用対象外の補正が混ざらない例

## Battle Scenarioテスト

### シナリオHarness

```text
runScenario({
  catalog,
  command,
  randomValues,
  expectedRules
}) -> SimulateBattleResult
```

Harnessは実際のFormation、Battle、Observationを使用し、HTTPとWorkerだけを通さない。次をまとめてassertできるようにする。

- 勝敗と終了理由
- 最終状態
- 特定イベントの順序とpayload
- 行動ID・SkillUse ID・親子関係
- 状態バージョンと差分
- RandomSource消費回数
- ルールIDの関連づけ

## PS連鎖テスト

PSは循環と順序の不具合が起きやすいため、専用のテスト群を持つ。

### 候補順

- 行動速度が異なる所有者
- 同速の味方と敵
- 同陣営同速の前列・後列
- 同じ列条件で絶対左列
- 同じユニットの複数PS定義順
- 先制攻撃候補を通常候補より優先し、先制攻撃同士へ全tie-breakerを適用すること
- 同時発動制限による1件選択

### 再確認

候補検出後、先行PSによって次を変化させる。

- 所有者が戦闘不能になる。
- PP不足になる。
- クールタイムが設定される。
- 発動条件が不成立になる。
- チャージ状態になる。

発動直前確認で候補が除外され、適切な診断理由を観測できることを確認する。

### ネスト

```text
元イベント
├─ PS-A候補
├─ PS-C候補
└─ PS-A発動
   └─ 子イベント
      └─ PS-B発動
         └─ 完全解決
   └─ PS-Aの残効果
└─ PS-Cへ復帰
```

イベントの `sequence`、`parentEventId`、`rootEventId`、候補スタック順を検証する。同じ解決スコープでPS-Aを再候補化しても2回目は発動しない。

## 効果期間テスト

### 行動単位

- 付与された当該行動終了では減らない。
- 他ユニットの行動では減らない。
- 対象自身の次回行動終了で1減る。
- 残1から0になった直後に失効する。
- 待機も対象自身の行動終了として減算し、戦闘不能で予約を除去されて行動しなかった場合は減算しない。
- 同じ種別の複数インスタンスが別々の残回数を持つ。

### ターン単位

- 付与ターン終了では減らない。
- 次のターン終了から1減る。
- 残1から0になった直後に失効する。
- 同時に複数失効した場合のイベント順が決定的である。

### 重複なし

- 効果量が異なる複数インスタンスを保持する。
- 最大効果だけ `isEffective` になる。
- 最大効果失効後、次点が同じ終了処理内で有効になる。
- `EffectExpired` → `EffectiveEffectChanged` → `CombatStatChanged` の相対順を確認する。各イベントから誘発されたPSイベントの割り込みは許容し、隣接だけを要求しない。

## イベントと状態差分

### イベント不変条件

- `sequence` が重複せず昇順である。
- 子イベントのsequenceが親より大きい。
- 同じ解決スコープでrootが一致する。
- 同じ行動・スキル解決でIDが一致する。
- 状態変更を所有するイベントが一つだけである。
- 内訳イベントに同じ差分を重複させない。
- TIMINGイベント後に親処理の前提を再検証する。
- DETAILEDからDIAGNOSTICを除いても状態履歴が変わらない。

### 独立した差分Reducer

テスト用に `StateTransitionResponse` を適用する小さなReducerを作る。productionのBattleObservationやResultAssemblerをそのまま呼び出さない。同じ不具合を期待値側でも再現することを防ぐ。

確認内容：

1. 初期 `stateVersion` が0。
2. 各差分でversionが1ずつ進む。
3. 前後versionが連続する。
4. `causedBySequence` が正規の内部イベント列に存在する主イベントを参照する。公開レベルでそのイベントが省略されることは許容する。
5. イベントの `stateTransitionIndex` が対応差分を指す。
6. 全差分適用後に最終状態と深い等価になる。
7. 差分を一件抜く、逆順にする、重複させると検証が失敗する。

### Snapshotの使用範囲

Snapshotを許容する対象：

- OpenAPI文書
- 小さく安定したイベント列の補助確認
- エラーレスポンスの代表例

Snapshotだけにしない対象：

- ダメージ計算
- 勝敗
- リソース消費
- PS順序
- 効果期間
- 状態差分復元

Snapshot更新時は仕様変更と単なる出力変化をレビューで区別する。

## Property／Modelテスト

ランダム生成にはテスト用seedを固定し、失敗時にseedと最小反例を記録する。再実行可能な形式でCIログへ残す。

### 主なProperty

- 任意の有効編成でHP、AP、PP、EX、シールド、期間が負数にならない。
- EXゲージがユニット固有最大値を超えない。
- 行動順比較器が反対称性、推移律、決定性を満たす。
- 入力配列順を変えても行動順結果が変わらない。
- 速度再計算でキューから行動済み者が復活しない。
- 速度再計算で予約済みAS／EXが変化しない。
- ターゲット距離が負にならず、マンハッタン距離の対称性を満たす。
- シールド吸収＋サブユニット吸収＋HPダメージが適用ダメージと一致する。
- リンク対象数を変えても各対象のリンクダメージ量が変わらない。
- 効果インスタンス追加・失効後も同一IDが重複しない。
- 初期状態へ全差分を適用すると最終状態になる。
- `COMPLETED` 後に状態が変化しない。

生成器はドメイン上有効な入力を基本とし、無効入力生成器は値オブジェクト・Command検証用に分ける。

## Applicationテスト

### Portのテストダブル

- `FakeBattleCatalog`
- `SequenceRandomSource`
- `FixedBattleIdGenerator`
- `ManualExecutionGuard`
- `InMemoryBattleObservation`

UseCaseの高速な単体テストではCatalogと実行保護を差し替える。Battleルール自体をmockして期待結果を捏造せず、正常系の主要ケースでは実Battleを使用する。

### 事前検証

- 複数のCommand違反を一度に収集する。
- Command違反時はCatalogやBattleを呼ばない。
- 不明なユニット・メモリーIDをpath付きで返す。

## 未実装Capabilityのテスト

現時点で保留仕様として隔離する `Q-*` Capability はない。未実装の `CAP_*` を必要とする定義は開始前に拒否する。

次をしてはならない。

- 仮の計算結果を期待値として固定する。
- 実際に分岐へ到達しなければ実行可能とする。
- production Catalogの説明文から未実装Capabilityを推測する。
- Capability IDを無視して通常効果として処理する。

Capability実装後は、拒否テストを新しいドメインルールの正常・境界テストへ置き換え、Capability集合と台帳を同時に更新する。

## Catalogテスト

### 構造

- manifestと各JSONがSchemaに適合する。
- ID、列挙値、整数・割合、期間が許容形式である。
- ファイルハッシュがmanifestと一致する。
- 未知プロパティを拒否する。

### 意味

- IDが種類内で一意である。
- すべての参照先が存在し期待する型である。
- 定義順が一意に決まる。
- PSトリガーイベントと条件式を解釈できる。
- 不許可の循環参照がない。
- 未実装Capabilityに `requiredCapabilities` が付いている。
- 実装済みCapabilityと定義の要求が整合する。

production Catalogの全定義を起動時検証と同じvalidatorで検査する。エラーfixtureでは一つの意図した違反だけを入れ、診断箇所までassertする。

## API契約テスト

### Catalog一覧

- 全Unit・Memoryをdefinition ID昇順で返す。
- 直接および推移的な未実装Capabilityを `unavailableCapabilities`へ重複なく返す。
- `selectable`が同じCatalog revisionの `SimulationPreflightValidator` 判定と一致する。
- `selectable: true`と空Capability、`selectable: false`と1件以上のCapabilityが対応する。
- Skill、EffectAction、Formula、Condition、triggeredEffectsの完全定義をresponseへ含めない。
- 200でCatalog revision由来ETagとCatalog用Cache-Controlを返す。
- ETag一致の `If-None-Match`で304、本文なしを返す。
- 一覧read modelをHTTP requestごとに再構築しない。

### リクエスト

- 1～5体、0～6メモリー、1～99ターンの境界
- 数値文字列、小数、`null`、未知プロパティの拒否
- 配置のcolumn・rowと共通座標変換
- 同一ユニット定義の重複許可
- Content-Type、Accept、本文上限
- `options` 省略時のDETAILED既定値

### レスポンス

- `schemaVersion`、`catalogRevision`、Battle ID
- 勝敗と終了理由の組み合わせ
- 初期・最終状態のスキーマ
- 公開イベントの連番と因果参照
- `stateTransitionIndex` と差分配列
- SUMMARYでも全状態差分を保持
- エラーのcode、path、ruleId、diagnosticId
- スタックトレースや内部パスがない
- `Cache-Control: no-store` とRequest ID
- Catalog一覧だけはETagと短いpublic cacheを返し、戦闘POSTの `no-store`と混同しない。

### CORS

- 許可したGitHub Pages originのCatalog GET、戦闘POST、preflightを受理する。
- 未許可originにCORS response headerを返さない。
- `Origin`なしの既存requestを拒否しない。
- browserから `X-Request-Id`、`Retry-After`、`ETag`を参照できる。
- 不正な `CORS_ALLOWED_ORIGINS`を起動時に拒否する。

### OpenAPI

- OpenAPI 3.0.3として検証できる。
- 全ルートと全ステータスにSchemaがある。
- 実際の代表レスポンスが生成Schemaへ適合する。
- 保存された文書を採用する場合は再生成差分がない。
- 互換性検査で意図しない破壊的変更がない。

## Worker・インフラテスト

### Worker境界

- DTOと結果が構造化クローンできる。
- HTTPオブジェクトやDomain Entityを渡さない。
- コンパイル済みESMワーカーをproductionと同じパスで起動できる。
- ワーカーごとにCatalogを一度だけ読み込む。
- 期待Catalogリビジョン不一致を拒否する。
- 複数リクエストのBattle状態が混ざらない。

### 容量と障害

- `concurrentTasksPerWorker = 1` を維持する。
- Pool満杯を `503 CAPACITY_EXCEEDED` にする。
- 利用者別上限を `429 RATE_LIMIT_EXCEEDED` にする。
- 協調期限を `504 EXECUTION_TIMEOUT` にする。
- クライアント切断で外側のタスクをキャンセルする。
- ワーカー異常終了後にタスクを自動再実行しない。
- 障害ワーカーが補充される。
- Graceful Shutdownの受付停止、待機、強制終了順を守る。

Worker結合テストは共有状態と実時間依存を避け、各テスト専用Poolを生成して必ずcloseする。

### Container・Cloud Run境界

- production containerをLinux amd64向けにbuildできる。
- containerがnon-rootで起動し、注入した `PORT`へ `0.0.0.0`でlistenする。
- image内のCatalog、コンパイル済みWorker、OpenAPIを解決できる。
- SIGTERM受信後8秒以内に受付停止、Pool終了、process終了を完了する。
- Cloud Run設定のminimum 0、maximum 1、concurrency 2、timeout 40秒が宣言値と一致する。
- GitHub ActionsのCloud Run deployが長期service account keyを使用しない。

## End-to-Endテスト

最小限の代表経路に絞る。

1. HTTPから最小Catalogの基本戦闘を完了する。
2. 同じCatalog revisionの一覧APIで選択不可の定義を確認し、戦闘POSTでも `422 UNSUPPORTED_RULE`になる。
3. GitHub Pages相当の別originから一覧GETと戦闘POSTを完了する。
4. PS連鎖と状態差分を含む戦闘を完了する。
5. 同一ユニット重複と最大編成人数を受理する。
6. 未実装Capabilityを含む定義をHTTP `422` で拒否する。
7. Worker Pool容量不足をHTTP `503` で返す。
8. タイムアウトをHTTP `504` で返す。
9. 初期状態からレスポンス差分を復元して最終状態と一致する。
10. Cloud Run revisionの公開HTTPS URLでlive、ready、Catalog GET、最小simulationを完了する。
11. deployed GitHub Pages originからCloud RunへのCORS preflightと実requestを完了する。
12. 新revisionのsmoke test失敗時に直前の正常revisionへtrafficを戻せる。

End-to-Endでは全109ルールを個別に再検証しない。失敗時に原因を切り分けられる小さな下位テストを正本とする。

## モジュール境界のリファクタリング（`#132`）

M6〜M8に備えたDomain内部モジュール境界の再編（`#132`、PR1〜PR5）は、既存挙動を一切変更しない前提のArchitecture refactoringである。次のテスト方針で非回帰を保証する。

### Module境界テスト

- Layer間の禁止依存を検証している既存のESLint `no-restricted-imports` テストと同様の方式で、[04\_境界づけられたコンテキスト.md](./04_境界づけられたコンテキスト.md)「モジュール依存規則」の禁止importをテストする（PR5で追加）。
  - `catalog` から `formation`／`battle` への import がないこと。
  - `formation/*` から `battle/*` への import が、明示的に許可した一方向（`battle/model`・`battle/targeting`・`battle/combat` の型・純粋計算のみ）以外に存在しないこと。
  - `battle/effects` から `battle/combat` への import がないこと（逆方向の `battle/combat` → `battle/effects` は許可）。
  - `battle/*` の下位モジュールから `battle/lifecycle` への逆依存がないこと。
  - `presentation` から `domain` への直接importがないこと（既存規則の維持確認）。
- 循環依存検査（`mise run check-circular` / `madge --circular`）を、移動後の新しいpathに対して実行する。

### 移動時のcharacterization

機械的移動（PR2: Domain、PR4: Application/Presentation）と責務分割（PR3: Domain、PR4後半のSchema/Route分割）は、それぞれ次を満たしてからマージする。

- 移動前後で、Unit／Scenario／Contract／Integration／E2E／Loadの各テストレベルの実行件数（`it`/`test`の件数）が減っていないことを確認する。テストファイルの移動に伴うimport更新だけを行い、アサーション内容は変更しない。
- 移動対象のファイルと、対応する `*.test.ts` は同一コミット内で一緒に移動し、実装内容の変更（責務分割・ロジック変更）は別コミットへ分離する。
- 責務分割PR（PR3、PR4後半）では、分割前の1ファイルに対する既存テストが、分割後も（importパスの変更以外は）そのまま通ることを確認する。新しい公開境界を追加する場合は、境界を検証する最小限のテストを追加してよい。

### 乱数・イベント順序の非回帰条件

- 固定 `SequenceRandomSource` を用いた既存の `SCN-BTL-*` シナリオが、移動前後で同じ乱数消費回数・消費順を維持することを確認する。
- 既存イベントの `type`・`sequence`・`parentEventId`・`rootEventId` の順序が、移動前後で変化しないことを確認する（スナップショットまたは既存アサーションで検証）。
- `StateTransition`（`stateVersionBefore`/`stateVersionAfter`/`stateDelta`）の内容が変化しないことを確認する。
- `initialState + stateTransitions = finalState` の不変条件が、移動後も成立することを確認する。
- OpenAPI文書、Catalogレスポンス、CORS許可設定、ETag挙動、エラーマッピングが移動前後で意味的に同一であることを契約テストで確認する（PR4対象）。

PR2〜PR5の各PRで、少なくとも次を実行する。
